# 查询、搜索与排序

> 类型：规范。有效字段、过滤、搜索、排序和分页语义以本文为唯一权威来源。

## 查询对象

所有列表查询都面向服务端已发布的 `query_publication_id`，它不可拆分地引用一个合法 `(catalog_revision, overlay_projection_revision)`，并在授权后的 Library/Source/CanonicalWork 范围内执行。客户端不能分别指定两类 revision；查询层从 Source-derived 事实和 `control.db` Overlay 生成有效字段，但不得把查询投影当作新的权威事实。

客户端提交查询意图，服务器负责字段解析、过滤、搜索、排序、分页和 capability 裁剪。不同客户端不得自行复刻排序键或在取回一页后重新排序。

## 查询快照的组成

`catalog_revision` 冻结 SourceWork/SourceCreator/SourceMedia、ContentBlob/FileLocation 和来源关系；`overlay_projection_revision` 冻结基于一个确定 Catalog revision 生成的 WorkProjection/CreatorProjection/MediaProjection、有效字段、可见性、SortKey、结构化关系与 FTS。服务端为每个合法组合创建 `query_publication_id`；查询、租约和 GC 只通过该 ID 选择组合。

Overlay 不按字段永久划分一致性类别，且具体某次查询实际依赖哪些字段不是一张全局静态表能表达的：服务端维护两层结构分离的机制。

### 字段能力注册表（静态）

`overlay.OverlayFieldCapabilities`（`internal/overlay/service.go`）登记每个 Overlay 字段**静态支持**哪些查询用途——是否可显示（`Display`）、可过滤（`Filter`）、可排序（`Sort`）、可搜索（`Search`）、影响可见性（`Visibility`）、解析为资源引用（`ResourceReference`），以及是否除 snapshot 值外还提供 control.db 当前值供即时展示（`LiveUserState`）：

| 字段 | Display | Filter | Sort | Search | Visibility | ResourceReference | LiveUserState |
| --- | --- | --- | --- | --- | --- | --- | --- |
| `titleOverride` | ✓ | | ✓ | ✓ | | | |
| `manualTags` | ✓ | ✓ | | ✓ | | | |
| `hidden` | | ✓ | | | ✓ | | |
| `customCoverMediaId` | ✓ | | | | | ✓ | |
| `favorite` | ✓ | ✓ | | | | | ✓ |
| `progress` | ✓ | ✓ | ✓ | | | | ✓ |

任何在这张表中声明 `Filter`/`Sort`/`Search`/`Visibility`/`ResourceReference` 能力的字段，写入时都必须先经过写后屏障（对应持久投影 Job 完成、进入新 `query_publication_id`）才能反映到查询结果——不因某次具体查询未使用该字段就跳过屏障，因为写入端无法预知未来会有哪些查询依赖它。`LiveUserState` 字段（当前 Favorite、Progress）在写后屏障之外**额外**提供 `GET /works/{workId}/overlay` 的即时读取，不需要等待新 publication。

### 查询 dependency set（动态、由 planner 生成）

具体一次查询实际依赖哪些字段，由服务端 `query.Service` 的 planner 在 `Search` 时按本次实际请求生成，不是把上表整体当成依赖集合。响应 `dependencySet` 是一个 `{field, role}` 列表，`role` 取值：

| role | 含义 |
| --- | --- |
| `predicate` | 字段作为结构化过滤谓词参与本次查询 |
| `ordering` | 字段作为排序依据 |
| `search` | 字段参与本次搜索召回/ranking（有非空搜索词时固定包含 title/creator/tag/filename） |
| `membership` | 字段影响默认集合成员资格但未被显式过滤（当前只有 Hidden 的默认隐式隐藏） |
| `resource` | 字段解析为另一个资源引用 |

例如：一次不带 `q`、不过滤 Favorite 的普通列表查询，`dependencySet` 只包含 `{overlay.hidden, membership}` 和 `{title, ordering}`；一次 `filter={"field":"overlay.favorite","op":"eq","value":true}` 的查询会额外包含 `{overlay.favorite, predicate}`；一次带 `q=xxx` 的查询额外包含 `{title|creator|tag|filename, search}` 四条。`dependencySet` 的规范化编码（`field:role` 排序后拼接）进入查询指纹，因此依赖集合不同的查询天然拥有不同指纹与游标身份。

响应同时携带静态 `liveUserStateFields`（当前 `["favorite","progress"]`），说明这些字段除 snapshot 值外还能通过 `GET /works/{workId}/overlay` 读取 control.db 当前值——用于向客户端解释"为什么刚保存的 Favorite 没有立即改变这个可能仍绑定旧 publication 的列表结果"。

写后屏障处理连续写、合并、superseded、失败、重试和服务重启：`work_overlays.query_watermark`/`projected_watermark` 追踪目标与已发布水位，`projection_status` 在 `pending`/`published`/`failed` 间迁移；旧 publication 在新投影完成前继续可读；`LiveUserState` 字段的 live 展示不受屏障影响，`snapshot` 判据在屏障完成前不改变当前页成员。

同一事实可以有“快照判定值”和“实时附加值”：响应 `works[].favorite`/`works[].progress` 是本次查询所在 publication 冻结的 snapshot 值（用于解释过滤/排序判据），而 `GET /works/{workId}/overlay` 返回 control.db 当前 live 值。两者不一致表示游标仍在旧 publication 中，不得用实时值在客户端删除、插入或重排当前页；刷新第一页后进入新 publication。

## v1 结构化查询

结构化过滤是一棵服务端权威 AST：恰好一个分支有效——`all`（AND）、`any`（OR）、`not`（取反）或叶子 `field`/`op`/`value`。请求携带 JSON 编码的树；服务端按下表已注册字段名与操作符校验，未知字段、未知操作符或值类型错误一律返回 `VALIDATION_ERROR`，不静默忽略，也不把任意 SQL、内部列名或 row ID 暴露给客户端。字段名与操作符集合是唯一权威来源，OpenAPI `filter` 参数只声明其编码为字符串，具体语义以本节为准。

| 字段 | 操作符 | 语义 |
| --- | --- | --- |
| `library.id` | `eq` | 按 Library 精确匹配 |
| `source.id` | `eq` | 按 Source 精确匹配 |
| `provider.id` | `eq` | 按规则声明的 Provider 精确匹配（关联 `source_works.provider_id`） |
| `creator.id` | `eq` | 按 CanonicalCreator 精确匹配，见下「Creator 合并与查询」 |
| `creator.role` | `eq` | 按 WorkCreator role 精确匹配 |
| `tag` | `eq` | 按标签精确匹配 |
| `media.kind` | `eq` | 按媒体种类精确匹配 |
| `media.locationAvailable` | `eq`（bool） | 按媒体位置是否可用（`location_status = present`）过滤 |
| `media.contentVerificationState` | `eq`（`located_unverified`\|`content_verified`） | 按内容确认状态过滤 |
| `overlay.favorite` | `eq`（bool） | 按 Favorite 的 publication snapshot 值过滤 |
| `overlay.hidden` | `eq`（bool） | 显式接管 Hidden 可见性，见下方专门说明 |
| `overlay.progress` | `eq`\|`lt`\|`lte`\|`gt`\|`gte` | 按 Progress 的 publication snapshot 值比较，value 必须落在 `[0,1]` |

`overlay.hidden` 语义：普通查询不显式引用该字段时，服务端隐式追加 `hidden = 0`（默认隐藏 Hidden 作品）。客户端显式在 filter 中引用 `overlay.hidden`（无论取值 `true`/`false`，无论是否被 `not` 包裹）即接管可见性语义、服务端不再叠加默认隐式条件，且该请求需要 `library.write` capability——只读 `library.read` 账户不能绕过默认隐藏可见性。这是刻意的保守统一门槛：对任意深度的 `all`/`any`/`not` 组合判断"最终是否只暴露非 Hidden 作品"等价于布尔可满足性问题，与其解析具体语义、留下遗漏空间，不如用统一门槛换取简单、无歧义、可审计的规则，避免 `not hidden=true` 与默认隐式条件产生不可解释的双重语义。

已知缺口（登记为明确缺口，不在过滤 DTO 中伪造空字段，见 [v1 实施计划](../指南/01-v1实施计划.md) 阶段 0–3 遗留表）：

- `WorkOrigin`（区别于 Provider 的独立来源模型）仍未实现，延后到领域 Schema 最终冻结门禁；
- 发布时间/发现时间没有对应的 Source-derived 或 Canonical 时间戳字段，未提供过滤；
- R-18/AI 等分类不作为专用字段，由规则产出的通用 `tag` 承载，用 `tag` 过滤即可覆盖；
- `Collection` 实体在当前领域模型中不存在，不提供过滤；
- Progress 当前只提供过滤，不提供排序（`按 Progress 排序` 仍是未实现的说明性用例，不在字段能力注册表之外伪造已支持）。

标题/CanonicalCreator/标签/文件名搜索见「搜索索引 v1」；已注册排序字段及升降方向见「排序协议 v1」。

## 搜索索引 v1

v1 使用 SQLite FTS5，并明确分离候选索引和权威复核文本：

```text
normalized_original_text  # NFKC + Unicode case fold，title\ncreator\ntag...\nfilename... 拼接，FTS 候选召回与子串复核来源
cjk_bigram_token_text     # 连续 CJK bigram 候选
latin_trigram_token_text  # 拉丁、数字和文件名连续 trigram 候选
structured_relations      # 标签、Provider、CanonicalCreator 等精确过滤

# 字段级 ranking/高亮投影（可从 Source-derived 事实与 Overlay 完全重建，不是新的事实源）：
search_title_norm         # 标题单值规范化文本
search_creator_norm       # Creator 单值规范化文本
search_tags_norm          # 各 tag 规范化后按 U+001F 分隔符连接的多值文本
search_filenames_norm     # 各文件名（仅 basename）规范化后按 U+001F 分隔符连接的多值文本
```

任何进入 `search_tags_norm`/`search_filenames_norm` 的单个取值不得包含 U+001F（`INFORMATION SEPARATOR ONE`）——不论来自用户手动输入（Overlay `manualTags`）还是规则/metadata 产生的 Tag/文件名，均在各自的权威落库入口（`overlay` Overlay 写入、`catalog.Store.Stage` Source-derived 事实落库）拒绝，返回结构化校验错误，不依赖"该字符不可能被输入"的假设。

候选 token 使用 AND 查询，命中后必须在规范化原文中验证连续性和顺序。token 不能单独证明命中，也不能直接给出高亮偏移。精确标签和 ID 使用关系表，不依赖全文分词。`normalized_original_text` 的拼接顺序不足以可靠恢复字段边界（标题允许包含字面换行），因此字段级 ranking 与高亮改为直接读取 `search_*_norm` 列，不再从拼接文本按换行位置猜测标题段。

## v1 搜索承诺

| 能力 | v1 语义 |
| --- | --- |
| 至少两个 CJK 字符的连续搜索 | 支持，bigram 候选 + 原文复核 |
| 拉丁/数字和文件名中缀 | 支持，trigram 候选 + 原文复核 |
| 大小写、全角半角 | NFKC + Unicode case fold 后匹配 |
| 标签/Provider/CanonicalCreator 精确过滤 | 使用结构化关系 |
| 单字符 CJK | 无强过滤时返回 `QUERY_TOO_SHORT`；不得静默全库扫描 |
| 拼音、日文汉字读音 | 仅搜索显式 `sort_reading`/语言字段；不自动猜测 |
| 模糊纠错和语义搜索 | 延后 |
| 高亮 | 从复核后的规范化原文映射；正式偏移算法须过黄金集 |

宽查询必须分页并尽早复核，不能把所有候选搬入应用层。

## Ranking 协议 v2

```text
rank_protocol_version = 2
```

只在请求携带非空搜索词时生效；无搜索词的浏览查询排序完全由「排序协议 v1」决定，不产生伪相关性排名。v2 从 v1 的"只有标题"单一 0–3 tier 改为标题/Creator/标签/文件名字段级 ranking：每个字段独立计算 `match_class ∈ {0,1,2,3}`（3=与规范化查询完全相等，2=以查询为前缀，1=包含查询子串，0=未命中；标签/文件名的"完全相等/前缀"按其中某一个具体取值判定，"子串"允许在连接文本中跨取值边界，是已记录的召回层简化），再按固定 `field_priority`（标题 3 > Creator 2 > 标签 1 > 文件名 0）合成：

```text
combined(field) = 0                              if match_class(field) = 0
combined(field) = match_class(field) * 10 + field_priority(field)   otherwise
rank_tier = max(combined(title), combined(creator), combined(tag), combined(filename))   # 0..33
```

该元组保证：完全匹配（任意字段）优于前缀（任意字段），优于中缀（任意字段）——`match_class` 是十位，任何字段的更高 `match_class` 都无法被另一字段的更高 `field_priority` 反超；同一 `match_class` 内按字段优先级排列；无搜索词时 `rank_tier` 恒为 0，与 v1 行为一致。最终排序为 `ORDER BY rank_tier DESC, sort_key, work_id`，同分按排序协议 tie-break，保持稳定。

`rank_protocol_version` 进入查询指纹与签名游标（`lastRankTier` 字段，合法范围从 v1 的 `[0,3]` 扩大为 `[0,33]`），tier 计算方式的任何变化都必须递增该版本号，使旧游标随之失效而不是静默产生不一致的续页顺序——v1 签发的游标在 v2 部署后一律失效。字段权重的具体数值（当前 3/2/1/0）在正式压力测试前保持 PRE_FREEZE，字段级结构与协议版本演进路径已冻结。

## 命中表达（`matches`）v1

v1 不再只返回 `titleHighlights`，改为通用、版本化的 `matches` 数组，每项 `{field, value, spans}`：

- `field` 取值为 `title`、`creator`、`tag`、`filename` 之一；
- `value` 是命中的原始显示值——`tag`/`filename` 是具体命中的**那一个**取值（不是整个列表），`filename` 只是 `path.Base` 之后的安全显示名，不泄露相对/绝对路径；
- `spans` 是该 `value` 内的命中区间列表，每个区间 `{start, end}`，单位是原文 **code point（Unicode 标量值/rune）偏移**（不是 UTF-16 code unit，也不是字节偏移），左闭右开（`end` 不含）；
- 偏移映射经过组合字符簇（基础字符与紧随其后的 Unicode 组合标记、emoji 变体选择符/ZWJ/肤色修饰符）聚合后计算，保证候选命中后的原文连续性复核不把一个组合字符簇拆开高亮；
- 同一字段可能出现多条（例如两个不同的 tag 分别命中，各自携带自己的 `spans`）；同一个 Work 的 Creator 命中若来自合并后的展示名，`value` 是合并后名称，与「Creator 合并与查询」一节的展示层重写路径一致；
- 单个 Work 的 `matches` 条目数与每个 `value` 的展示长度均有服务端截断（当前分别为 8 条与 512 rune），防止病态数据无界增长；`spans` 必须先针对完整原文计算，再按截断后实际返回的 `value` 裁剪：起点落在截断边界之后的命中整体丢弃，跨越边界的命中把 `end` 收紧到截断长度，保证每个返回的 span 都满足 `0 <= start <= end <= runeCount(value)`，不越界指向 `value` 之外的字符；
- 无搜索词时不返回 `matches`。

## Total 语义 v1

```text
total_protocol_version = 1
```

`total` 字段区分三种模式，避免普通列表路径执行无上限全库 `COUNT`：

| mode | 语义 |
| --- | --- |
| `exact` | 命中行数不超过服务端预算，`value` 为精确计数 |
| `lower_bound` | 命中行数超过预算，`value` 为预算值（真实命中数 ≥ 该值） |
| `omitted` | 客户端通过请求参数显式跳过统计，`value` 缺省 |

预算数值本身在正式压力测试前保持 PRE_FREEZE；`total_protocol_version` 与三态结构是冻结的兼容演进基线，未来调整预算策略不需要破坏性变更。

## Creator 合并与查询

CanonicalCreator 合并是 `control.db` 中可撤销的展示层重定向：`canonical_creators.merged_into` 记录当前有效根，`work_creator_relations.creator_id` 永远保留合并发生时的原始 Creator ID，不因合并或撤销被改写。

- 按 `creator.id` 过滤时，服务端把输入 ID（可以是合并根，也可以是已被合并的旧 ID）解析为完整等价组（根与全部直接/间接被并入成员），命中该组任一成员的作品；
- `GET /api/v1/creators/{creatorId}` 返回 `effectiveId` 作为稳定重定向：查询已被合并的旧 ID 时明确得到其当前根，而不是静默 404 或返回过期展示；
- 撤销合并只清除 `merged_into` 并重新排队投影，不触碰 `work_creator_relations`，因此过滤语义随撤销立即、完整地恢复；
- 多级合并链（A 并入 B，B 又并入 C）在等价组解析时被折叠：查询 A、B 或 C 任一 ID 都命中同一组有效作品；
- 展示层的 `work_projections.creator` 文本与 FTS 索引由投影阶段的 `ApplyCreatorMerges` 重写，与全量 Catalog 重建走同一路径，因此删除重建 `catalog.db` 后过滤与展示语义保持一致。

## 排序协议 v1

```text
sort_protocol_version = 1
text_key = natural_chunks(unicode_case_fold(NFKC(text)))
nulls = last
SQL = ORDER BY rank_tier DESC, sort_key, work_id
```

- 数字块按去前导零后的有效长度和值排序，再以原数字长度确定性打破等值；
- 非数字块使用稳定 Unicode/UTF-8 回退，不依赖系统 locale；
- `work_id` 是独立 SQL tie-break，不拼入 sort key；
- null 永远排在非 null 之后，升降序的具体编码必须在黄金语料中固定；
- 拼音和日文读音仅来自 metadata、用户输入或版本化语言模块提供的 `sort_reading`；缺失时回退基础协议；
- 协议升级必须更改 version，使旧游标过期。

时间排序必须先解析为明确 instant，并保留原始时间和解析规则版本。不得按未规范化时间字符串或 OS 本地时区排序。

## 快照分页

Gallery API v1 对有序列表承诺 revision 快照一致。签名游标至少绑定：

```text
query_fingerprint
sort_protocol_version
rank_protocol_version
query_publication_id
authorization_scope_hash
last_sort_key
last_rank_tier
last_canonical_work_id
issued_at
lease_id / expires_at
```

`query_fingerprint` 的输入同时覆盖结构化过滤树的规范化 JSON 编码，过滤条件变化与搜索词、排序方向变化一样使旧游标续页失败为 `CURSOR_EXPIRED`。

后续页使用与 SQL 相同的 keyset 元组；示意为 `(sort_key, canonical_work_id) > (?, ?)`，实际比较符号随方向确定。`query_publication_id` 是服务端签名游标内唯一可选择快照的句柄，不是 SQLite row ID；裸 `catalog_revision`/`overlay_projection_revision` 即使出现在诊断响应中也不能作为查询参数。

| 变化 | 行为 |
| --- | --- |
| 新 query publication 发布 | 租约内继续旧 `query_publication_id`；新查询才进入新 publication |
| 当前 revision 中排序字段变化 | 不会发生；revision 不可变 |
| 查询/过滤改变 | `CURSOR_EXPIRED` |
| 授权范围、Session 或 Library 状态改变 | `CURSOR_EXPIRED` |
| 排序协议升级 | `CURSOR_EXPIRED` |
| query publication 已 GC、内部配对失效或租约过期 | `CURSOR_EXPIRED` |
| 游标签名/结构无效 | `CURSOR_INVALID`，不泄露内部值 |

服务器不为长期离线游标无限保留 revision。客户端收到 `CURSOR_EXPIRED` 后必须从第一页刷新，不得猜测游标内部字段。

## 一致性和缓存

- 响应必须带查询使用的 `query_publication_id` 和排序协议版本；可附带两类 revision 作为只读诊断信息；
- ETag/缓存键包含查询指纹、`query_publication_id` 和授权范围；带实时 `userState` 的私有响应还必须包含用户状态版本或禁止共享缓存；
- 搜索文档、有效字段、结构化关系和 WorkProjection 必须与同一 revision 元组发布；
- 当前查询依赖的 Overlay 只通过新的 query publication 生效，不允许一页内混用新旧有效字段；未进入本次 `dependencySet` 的字段不得反向参与集合、排序、搜索、过滤或可见性；
- 客户端本地筛选只能用于纯展示，不得改变服务端声明的总集合语义。

## Filter JSON 严格解析

`ParseFilter` 拒绝任何顶层尾随内容。判定方式是对同一个 `json.Decoder` 再调用一次 `Decode` 并要求恰好得到 `io.EOF`：单独调用 `Decoder.More()` 不足以可靠判定顶层尾随 JSON——当尾随字节恰好以 `}` 或 `]` 开头时（例如 `{...}}`、`{...}]`），`More()` 会把它误判为"当前容器的结束符"而不是额外数据，从而放过本应拒绝的输入。这一区别只影响服务端解析实现，不改变 filter 语义本身。

AND/OR 子项在语义上可交换，但当前查询指纹仍保留客户端提交的原始子项顺序（不做规范化重排）——等价查询可能因为子项顺序不同而产生不同的游标/缓存身份。这是已记录的简化，继续标记 `PRE_FREEZE`；是否安全实现"按 canonical JSON 排序 all/any 子项"仍待评估（需要保留重复谓词与嵌套语义），不得声称 canonical query 已最终冻结。
