# v1 实施计划

> 类型：实施指南。开发顺序、首条垂直切片和 v1 交付边界以本文为唯一权威来源。

## 实施策略

先建立一条能证明核心边界的纵向链路，再扩展平台和 UI。不得先写完整前端、完整表结构或全部平台规则，最后才验证数据所有权和发布一致性。

建议源码职责布局如下，具体包名可调整：

```text
gallery/
  cmd/                 # galleryd, galleryctl
  domain/              # 实体、值对象、端口
  application/         # 用例与事务边界
  rules/               # Schema、编译器、Rule IR、Trace/Impact
  control/             # 不可重建事实仓储
  catalog/             # revision、查询投影、搜索
  scanner/             # 发现、候选关联、staging 构建
  jobs/                # 持久任务与有界调度
  media/               # Range、ETag、DerivedAsset
  transport/           # HTTP、OpenAPI、WebSocket
  platform/            # OS 适配器
  web/                 # Web/PWA
  shell/               # 可选薄壳
```

这只是模块边界，不冻结类名、函数名或表名。

## 阶段 0：契约骨架

先交付：

- 领域 ID 和 Source/CanonicalWork/CanonicalCreator/CanonicalMedia/ContentBlob/FileLocation 的最小值对象；
- `control.db` / `catalog.db` 的 Schema 升级框架、备份接口和独立生命周期；
- OpenAPI、错误 code、WebSocket 信封、排序协议和游标的可执行 Schema；
- 规则包 JSON Schema、编译器版本和 Gallery CEL Profile；
- AppDirs 与媒体根写入守卫；
- 可注入的时钟、ID、文件系统和进程端口，便于确定性测试。

完成条件：规范中的实体和错误可由契约测试引用，但尚不冻结最终物理表结构。

## Walking Skeleton

> 实施状态（2026-07-16）：已完成。一个作品/一个媒体的仓库内合成只读 Source 已贯通 Personal 配对、Library/Source、RuleVersion/SourceRuleBinding、持久 Job、完整 SHA-256、Canonical Binding、staging publication、REST、媒体 HEAD/GET/Range、WebSocket 和服务重启恢复。证据与局限见 [验证记录 EV-13](../证据/验证记录.md#ev-13-walking-skeleton-e1--e4e5)。

使用 Test-Bench 内仅含一个作品和一个媒体的合成只读 Source，尽早打通最小纵向链路：

1. 启动 `galleryd`；
2. Personal 模式建立最小管理 Session；
3. 创建一个 Library 和一个 Source；
4. 创建 SourceRuleBinding，绑定并编译最小 RuleVersion/RuleParameters；
5. 扫描一个 SourceWork 和一个 SourceMedia，对媒体完成首次完整 SHA-256；
6. 构建最小 Catalog revision 和匹配的 Overlay projection revision，在短事务内创建 `query_publication_id` 并切换活动 publication；
7. 通过生成的 OpenAPI 客户端和 REST 获取一个 CanonicalWork，响应携带 `query_publication_id`；
8. 通过稳定 CanonicalMedia ID 执行一次 HEAD 和单区间 Range；
9. 通过 WebSocket 收到 Job completed，并能用 HTTP Job snapshot 复核；
10. Web、CLI 或最小测试客户端全程不访问数据库或后端内部包。

Walking Skeleton 只证明模块依赖方向、两库基础生命周期、API-first、OpenAPI 生成、AppDirs、最小扫描 publication 和客户端边界。它暂不要求完整 FTS、快照分页、Overlay 写入/Catalog 重建、多个强杀点、完整 RuleImpact、多平台或真实大库性能，不能据此冻结物理 Schema 或完整 API。

功能可以精简，基础模型不能临时替代：必须使用正式 `control.db`/`catalog.db` 迁移框架、Canonical/Source 稳定 ID、SourceRuleBinding、ContentBlob 完整哈希身份、`query_publication_id`、正式错误信封/OpenAPI DTO 和 capability 检查。禁止用以后需要替换的内存数据库、路径主键、临时 DTO、匿名管理员或客户端直连数据层“先跑通”。Walking Skeleton 不冻结全部表和字段，但已选中的身份、所有权、授权与 publication 契约只能通过兼容迁移扩展，不能在 Architecture Proof 时推翻。

## Architecture Proof Slice

> 2026-07-16 状态：本切片的合成正确性门禁已经通过，证据见 [验证记录 EV-14](../证据/验证记录.md)。第一版物理 Schema 与完整 API 暂不冻结；下一步进入阶段 1 的 Canonical/Binding 人工修复和多 Source 领域闭环，同时保留百万级查询、Reference Performance 和平台退化门禁。

Walking Skeleton 通过后，以下完整切片随阶段 1–4 逐步补齐，并作为数据库 Schema 与查询/API 契约冻结前的架构验收：

使用 Test-Bench 内的合成只读 Source，完成以下端到端路径：

1. Personal 模式启动 `galleryd`，一次性配对取得管理 Session；
2. 创建一个 Library 和一个 Source，以 SourceRuleBinding 绑定一个最小 RuleVersion 与参数；
3. validate/compile/Dry Run，得到 Rule IR、Trace 和 Impact；
4. 创建 Scan Job，分批构建 staging revision；
5. 在短事务内发布 SourceWork/SourceCreator/SourceMedia、ContentBlob/FileLocation，以及匹配的 WorkProjection/CreatorProjection/MediaProjection、SortKey 和 FTS；Canonical 实体仍以 `control.db` 为权威；
6. 通过 OpenAPI 浏览、过滤、搜索、排序，并以绑定合法 revision 组合的 `query_publication_id` 快照分页；
7. 通过稳定 CanonicalMedia ID 执行 HEAD、Range、ETag 和条件请求；
8. WebSocket 观察任务并在断线后从 HTTP snapshot 恢复；
9. 设置标题 Override 和 Favorite，等待 Overlay projection 发布；删除并重建 `catalog.db`；证明稳定 Binding、Override、Favorite 和固定 Blob 引用恢复；
10. 在构建中、发布前和发布后强杀进程，验证旧/新 revision 与 Job reconciliation。

Architecture Proof 通过条件：媒体 Source 前后清单一致；没有绝对路径或 Catalog 内部 row ID 出现在公共 API/永久 control 事实；所有失败均有稳定 code；客户端没有直接数据库或内部包依赖；查询全程不混合 Catalog/Overlay/FTS 代次。

本轮实现已经证明：正式规则包可经 API 完成 validate/compile/Dry Run/Trace/Impact；查询通过服务端签发的 `query_publication_id` 固定 Catalog/Overlay/FTS 代次并以签名 cursor 续页；query-affecting Overlay 使用持久 Job 重投影；删除 `catalog.db` 后 Canonical Work/Creator/Media、稳定 Binding 和用户事实可恢复；Blob/Location/DerivedAsset 与旧 revision 受读取租约保护；八个独立子进程强杀点均能恢复或以稳定错误收敛。该结论只提升 Architecture Proof 的 Correctness 状态，不替代下面各阶段的完整领域、性能、安全和平台门禁。

冻结评估采用分层结论：forward-only migration 的事务/checksum 规则、`query_publication_id` 合法 revision 组合语义、WebSocket v1 信封、RuleVersion/Rule IR 三类身份以及 Media/Blob/Location 稳定引用可以作为兼容演进基线；`control.db`/`catalog.db` 最终物理约束、完整 REST 资源、排序/过滤全集和 cursor 内部格式仍不得冻结。阻塞项包括 CanonicalCreator 合并/撤销与误绑定修复、超大文件持久哈希 Job、100 万/1000 万正式查询语料、ranking/highlight/total 的 API 表达、1%/10%/50% publication 参考性能、HDD/NAS/磁盘满退化以及目标平台安全门禁。

## 阶段 1：领域和数据所有权

- 完成 Library/Source/Provider、SourceWork/SourceCreator/SourceMedia、CanonicalWork/CanonicalCreator/CanonicalMedia、Binding 和 Overlay 的仓储；
- 实现 CanonicalCreator 合并/撤销、SourceWork inactive/orphaned 和人工 Binding review；
- 实现 CanonicalMedia/ContentBlob/FileLocation/DerivedAsset 身份与完整 SHA-256 确认；
- 建立 control 备份/恢复和 Catalog 全量重建测试；
- 通过 Schema 冻结门禁后再确定唯一约束。

## 阶段 2：规则闭环

- 实现有限原语、CEL Profile、编译缓存和 Rule IR；
- 支持规则包版本、参数复用、未知扩展无损往返和原子保存；
- 建立表单、文本编辑、Dry Run、Explain、Impact、版本 diff 和回滚；
- 将真实样本中验证过的三类目录/metadata 形态转为内置示例，不写平台分支；
- 对身份规则变化强制 Binding review，对展示变化只重投影。

## 阶段 3：扫描、任务和 Catalog

- 实现完整 staging revision、短发布、游标租约、候选清理和 GC；
- 持久 Job、attempt、取消、重试、进度序号和跨库 reconciliation；
- Watcher 仅作提示，周期校验负责收敛；
- Blob 哈希、扫描、缩略图和外部工具使用独立有界池；
- 在 SSD、HDD 和离线/网络挂载夹具上记录构建及维护窗口。

## 阶段 4：查询和媒体

- 完成结构化过滤、FTS5 bigram/trigram、原文复核和搜索分页；
- 冻结排序黄金语料、null 语义和游标签名；
- 完成有效字段投影、`query_publication_id`、独立 `overlay_projection_revision` 的构建/发布/租约，以及 Overlay 写后屏障和失败恢复；
- 完成 CanonicalMedia/ContentBlob 的 Range、ETag、离线错误和 DerivedAsset Job；
- 明确宽查询 total/highlight/ranking 后冻结 API Schema。

## 阶段 5：账户、安全和多客户端

- Personal 配对、Host/Origin/Fetch Metadata/CSRF、Session/WS 吊销；
- LAN Owner 初始化、Argon2id、本地账户、API Token 和 Library grant；
- available/effective capabilities 在 API、Web、CLI 全链一致；
- 分享 scope、过期和吊销；
- 威胁模型中的路径、恶意 metadata、恶意媒体和限速门禁。

## 阶段 6：Web/PWA

- 浏览、搜索、CanonicalWork、CanonicalCreator、CanonicalMedia 查看、Overlay、Job、规则和管理最小页面；
- headless 可访问组件 + 语义令牌；
- 键盘/焦点/ARIA、reduced motion、对比度和移动触控；
- 所有 UI 状态从 API/Schema/effective capabilities 产生；
- 普通浏览器完成所有业务闭环后，才开始壳集成。

## 阶段 7：平台与发行

- Windows 真实运行、升级、崩溃恢复、junction/reparse、CredentialStore 和签名；
- Linux 原生 ext4 核心门禁；macOS 和 Docker 按目标支持级别执行；
- SMB/NAS/UNC 独立门禁；
- Wails 与 Tauri 对照，决定是否将 Wails 从有条件接受提升；
- 程序/数据分离、备份恢复、版本升级、SBOM 和依赖安全检查。

## 不阻塞开工的不确定性

以下问题只要保留端口和版本字段，不阻塞阶段 0/垂直切片：最终前端框架、Wails/Tauri、macOS 是否列入 v1、精确 total 的 UI、语言读音模块、插件运行时、压缩包/PDF 支持。

以下问题会阻塞相关冻结：Binding 唯一约束阻塞领域 Schema；`query_publication_id` 对内部合法 revision 元组的绑定、Overlay 写后屏障的物理实现门禁阻塞查询 Schema；游标租约和 total/highlight 阻塞完整 API；平台文件身份和只读句柄复核阻塞正式平台发布。这些不阻塞按正式基础模型启动 Walking Skeleton。

完整门禁见 [测试与发布门禁](02-测试与发布门禁.md)。
