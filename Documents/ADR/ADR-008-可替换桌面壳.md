# ADR-008 可替换桌面壳

- 状态：有条件接受
- 日期：2026-07-16
- 规范：[跨平台与客户端](../规范/09-跨平台与客户端.md)

## 问题

Gallery 需要目录选择、托盘、自启、系统凭据和后端生命周期等桌面能力，但业务 UI 已由 Web/PWA 提供。必须决定桌面壳是否成为产品架构依赖，以及当前候选能否冻结。

## 决策

桌面壳是可替换 adapter，不是业务客户端的特殊后门。`galleryd` 可独立运行，普通浏览器具有完整业务能力；壳只处理进程/descriptor、目录选择、托盘、通知、自启、打开外部目标、CredentialStore bootstrap 和电源事件，并通过正式 API 使用业务能力。

Wails 是当前 Windows 优先原型候选，但在门禁完成前保持可替换，不能成为后端包、数据库或 Session 的依赖。Tauri 是主要对照；无壳服务模式始终受支持。

## 理由

- 单套 Web UI 降低多端维护成本；
- 独立后端进程契约已在 Windows/WSL 验证动态端口、单实例、健康和崩溃重连；
- Wails Windows 最小壳已构建和启动，包体/运行形态符合薄壳方向；
- 现有证据没有覆盖跨平台 GUI、更新、签名和全部系统集成，不能将候选提升为不可替换依赖；
- 证据见 [EV-10](../证据/验证记录.md#ev-10-后端独立与桌面壳e1--e4e5)。

## 替代方案

- **只提供浏览器**：最简单，但目录选择、托盘、自启和用户友好生命周期不足。
- **Electron**：生态成熟但捆绑运行时较大，当前没有必须使用 Node/Chromium 的业务理由。
- **Tauri**：体积小、系统集成强，是有效对照，但引入 Rust 工具链和双语言维护。
- **原生/跨平台 UI 重写业务界面**：造成两套 UI 和授权/契约漂移。
- **壳直连数据库/内部服务**：耦合实现、绕过安全和破坏无壳模式，明确拒绝。

## 影响

- descriptor 必须版本化并受当前 OS 用户权限保护；
- 进程 ownership、graceful shutdown、orphan policy、崩溃退避和协议不兼容需要正式契约；
- 壳测试分为系统桥接和同一 Web UI，两者不复制业务测试；
- 后端升级、数据升级和 API 兼容不能由壳私自处理；
- v1 即使只在 Windows 发布壳，核心仍保持无壳和跨平台能力。

## 提升为“接受”的门禁

- 接入真实 `galleryd` 的启动、连接、停止、单实例和崩溃退避；
- 目录选择、托盘、自启、CredentialStore 和电源事件完成；
- Windows 签名/更新/覆盖升级与数据保留通过；
- Linux/macOS 若宣称支持，则完成 GUI 运行、托盘、签名/公证和 AppDirs；
- 与 Tauri 对照包体、内存、构建、更新、安全边界和长期维护成本；
- 后端独立运行的完整 Web/CLI 回归持续通过。

## 重新审议条件

完成以上门禁后可将 Wails 提升为当前接受实现；若 Tauri 或无壳模式在维护、安全或发行上明显更优，则替换候选，不影响其他 ADR。若桌面系统能力被产品验证为非必要，可取消壳而保留服务模式。
