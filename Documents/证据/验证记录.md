# 验证记录

> 类型：验证证据。本文只保存仍支撑当前决策的聚合结果，不定义产品规范。原型源码和机器可读结果保留在 `Test-Bench/cleanroom-lab-real`；较早的技术对照保留在 `Test-Bench/cleanroom-lab`。

## 证据等级

| 等级 | 含义 |
| --- | --- |
| E1 合成单元/契约 | 可重复验证语义和边界，不代表真实规模或平台 |
| E2 合成性能 | 在记录机器和版本上的比较，可支持方向，不直接构成普遍 SLA |
| E3 真实只读样本 | 验证真实目录/metadata 形态；必须用前后 guard 证明零写入 |
| E4 目标平台运行 | 在目标 OS/文件系统实际执行核心或发行流程 |
| E5 交叉编译/推断 | 只证明可构建或提供估算，不代表运行支持 |

执行日期为 2026-07-16。结果来自本机固态存储、当时的 Go/SQLite/工具版本；正式实现必须在 CI 和目标硬件重跑。

## EV-03 媒体身份（E1 + E3/E4）

Windows 和 WSL 夹具覆盖改名、同卷移动、复制、硬链接、symlink、同作品重复和跨 Work 重复。人工构造两个 192 KiB 文件：大小及首尾 64 KiB 完全相同，中间不同；快速指纹相同，完整 SHA-256 不同。

真实媒体各抽取 1,500 个完整内容哈希样本，共发现 83 组确认重复；未观察自然快速碰撞，但人工夹具已证明采样签名不能作为权威键。

支撑：CanonicalMedia、ContentBlob、FileLocation、DerivedAsset 分离；ContentBlob 由完整 SHA-256 确认；FileID/路径/采样仅作候选。

局限：跨卷实际移动、SMB 重挂载、inode 复用、稀疏文件和 junction/reparse 尚未运行。

## EV-04 Catalog 提交（E2）

同一百万 Source-derived/查询投影、关系、SourceMedia/ContentBlob/FileLocation 和 FTS 合成数据，对 1%/10%/50% 变化比较两种可恢复模型：

| 模型 | 变化 | 长构建 | 发布事务 | 发布前空间 | GC/VACUUM 后空间 |
| --- | ---: | ---: | ---: | ---: | ---: |
| generation-delta | 1% | 104 ms | 960 ms | 468.0 MiB | 213.6 MiB |
| generation-delta | 10% | 819 ms | 4.701 s | 487.5 MiB | 218.0 MiB |
| generation-delta | 50% | 4.017 s | 20.774 s | 577.7 MiB | 237.0 MiB |
| staging-snapshot | 1% | 40.730 s | 4 ms | 610.9 MiB | 214.5 MiB |
| staging-snapshot | 10% | 40.526 s | 3 ms | 610.7 MiB | 213.9 MiB |
| staging-snapshot | 50% | 40.431 s | 8 ms | 604.1 MiB | 211.1 MiB |

两种模型在构建和回滚探针中都保持旧快照可读，Catalog 与 FTS 同一 Catalog revision。staging 的单次 publication 观测为 3–8 ms 且未随变化比例增长，因此支撑默认 staging snapshot + publish pointer 的方向；它不等于已经通过正式 P95 门禁。

局限：本机 SSD；staging 的完整构建、约 604–611 MiB 峰值、约 14–15 秒 GC 和约 6.5 秒 VACUUM 需要在 HDD 重测；该原型未覆盖独立 Overlay projection revision，其配对发布和重投影性能仍需 Architecture Proof 验证。

## EV-05 搜索（E2）

实现 CJK bigram、拉丁/文件名 trigram 和规范化原文复核。Bleve 对照使用默认字段映射、CJK analyzer、MatchQuery、Scorch/BoltDB；没有为 filename infix 设计专用 n-gram analyzer 或 query：

| 指标 | FTS5 100 万 | FTS5 1000 万 | Bleve 100 万 |
| --- | ---: | ---: | ---: |
| 构建时间 | 28.704 s | 333.083 s | 21.087 s |
| 索引/文档体积 | 285.6 MiB | 2.789 GiB | 209.1 MiB |
| 选择性 CJK 分页 P95 | 0.725 ms | 1.838 ms | 39.033 ms，且召回过宽 |
| 选择性文件名分页 P95 | 1.665 ms | 2.759 ms | 当前 analyzer/MatchQuery 下宽查询 0 命中 |
| 宽 CJK 前 200 条 P95 | 1.044 ms | 1.848 ms | 17.981 ms |

双字/多字中日文、英文、全角半角、大小写、数字、假名和文件名中缀的候选召回与复核准确率均为 1.0。千万样本宽查询若把全部候选搬到 Go 需约 9.4–14.8 秒，证明必须分页和尽早复核。

支撑：v1 FTS5 限定能力。当前 Bleve 原型在所采用的 analyzer 与 query 配置下未通过 filename infix 门禁，并且会引入独立索引与 Catalog revision 的协调成本，因此不作为 v1 默认引擎；该结果不能扩大为“Bleve 本身不支持文件名中缀搜索”。

局限：合成短文本；单字符 CJK、拼音、模糊、正式排名/高亮和 Bleve 千万级未验证。

## EV-06 排序协议（E1）

黄金样本验证 `file2 < file10`、全角/半角和大小写等价、emoji 数字自然序、nulls last、相同键下独立 ID tie-break。Windows、Linux、macOS 目标使用同一 Go 生成器；Linux/macOS 此处是同源构建证据，不是完整目标平台 UI 对拍。

支撑：排序协议 v1 和 `ORDER BY sort_key, work_id`。

局限：完整 grapheme、null 降序和正式 `sort_reading` 语料仍需扩充。

## EV-07 规则与 CEL（E1 + E3）

自建原语、CEL 和 Starlark 均能表达最小规则目标；可分析性、安全性和 UI 生成能力依次下降。CEL 原型编译并限成本执行 4 个条件/集合谓词，拒绝不存在的 host 函数；真实规则结果见 EV-02。

支撑：规范 JSON + JSON Schema + 有限原语 + 受限 CEL，配置内不执行 Starlark/JavaScript。

局限：未实现插件/WASM；这两项已延后，不阻塞 v1。

## EV-08 跨库、游标和 Personal 契约（E1）

原型验证：

- Catalog 已发布而 Job 仍 publishing，可修正 completed；
- Job completed 而无 publication，进入 needs_repair；
- 游标对 query、sort version、内部 snapshot revision、auth scope、GC 和 HMAC 篡改产生预期分支；
- Personal 匿名非管理员、首次配对、配对码复用拒绝、Host/Origin 拒绝、CSRF、capability 交集、多标签共享和 Session/WS 吊销均通过。

支撑：跨库 Saga、快照分页和 Personal 浏览器配对。

局限：原型验证了游标绑定内部 snapshot revision，尚未单独验证公共 `query_publication_id` 对合法 Catalog/Overlay revision 元组的封装；这属于查询 Schema 冻结门禁。Sec-Fetch-Site 浏览器 E2E、各浏览器 Cookie 细节和真实 CredentialStore 也尚未完成。

## EV-09 路径和跨平台核心（E1 + E4/E5）

常见 traversal、绝对/盘符/UNC、Windows 保留名、尾点/尾空格、NUL 和 symlink 逃逸共 14/14 按预期拒绝。相同核心探针在 Windows 11/NTFS 和 WSL Debian/DrvFS 实际运行；Linux amd64 与 macOS amd64 完成交叉编译。

Windows/DrvFS 验证大小写行为、NFC/NFD 名称并存、hardlink/symlink、长路径夹具、平台 FileIdentity 和工具发现。WSL 使用 DrvFS，不能代替原生 ext4；macOS 只可构建；Docker daemon 未运行。

支撑：跨平台核心 + 适配层、路径不是身份、Watcher 只作提示。

局限：ext4、APFS/FSEvents/Keychain、junction/reparse、SMB/NAS、Docker 和 HDD 均待门禁。

## EV-10 后端独立与桌面壳（E1 + E4/E5）

独立 `galleryd` 模拟器在 Windows 和 WSL Linux 验证 loopback 动态端口、原子 descriptor、健康检查、单实例锁、被杀后重启和重连；实现不引用 Wails。

Wails 2.13.0 Windows 最小壳构建约 28.4 秒，产物约 11.0 MiB，启动后 3 秒工作集约 27.4 MiB。它只验证壳体积和启动，不含真实后端、托盘、自启、目录选择、更新或签名。

支撑：后端独立、壳可替换；Wails 仅有条件接受。

## EV-11 后端发行技术对照（E2/E5）

早期最小发行原型中，Go 静态二进制约 6 MiB、启动约 154 ms；含 SQLite/Bleve 的组合约 6–16.6 MiB，Linux 交叉编译产物约 6.25 MB。ASP.NET self-contained 未裁剪约 98.8 MiB、启动约 686 ms；激进裁剪出现运行时 `TypeLoadException`。

支撑：在本地优先、单文件、跨平台和小团队维护目标下选择 Go。

局限：最小原型不代表完整产品包体；Go idle 内存读数受环境干扰，未作为决策依据；Docker 镜像大小只是静态二进制 + scratch 的推断。

## EV-12 阶段 0 正式契约骨架（E1 + E4/E5）

正式仓库使用 Go 1.26.5 和 `modernc.org/sqlite` 1.53.0 建立第一条可运行工程基础。在 Windows 10.0.28000/NTFS 上，`CGO_ENABLED=0 go test ./...`、`go vet ./...` 和 `go build ./cmd/...` 通过；真实 `galleryd` 进程以动态 loopback 端口启动，创建独立 WAL `control.db` / `catalog.db`、原子 runtime descriptor，并由生成的 `galleryctl` OpenAPI client 获得两库 `ok` 后优雅停止。合成 Source 的 SHA-256 在启动前后相同；AppDirs/Source 重叠在目录创建和数据库初始化前失败。

数据与契约测试覆盖两库 migration 版本/历史 checksum、防部分 migration、control 独立备份与完整性检查、带实体前缀 UUIDv7、完整 SHA-256 Blob 引用、HMAC 游标篡改/过期、错误/WS/规则 JSON Schema、匿名非管理员和匿名 WS 拒绝。`govulncheck` 1.6.0 对当前可达调用未发现漏洞。

同版工具链在 WSL Debian/DrvFS 上执行 `go test -race ./...` 通过。Windows 本机 race 二进制在测试代码运行前统一以 `0xc0000139` 退出；检查确认该系统的 `kernel32.dll` 不导出 Go race runtime 静态导入的 `WaitOnAddress`，而 `kernelbase.dll` 仍导出它。因此这是当前 Windows build 与 Go race runtime 的环境兼容限制，不记为代码竞态通过；Linux CI 保留 race 门禁。

局限：该证据仅覆盖阶段 0 合成正确性和当前 Windows/WSL 运行，不包含 Walking Skeleton 的配对、扫描、完整哈希 publication、媒体 Range 或 Job WebSocket；WSL DrvFS 仍不代表原生 ext4，也不提升任何平台支持级别。

## EV-13 Walking Skeleton（E1 + E4/E5）

正式仓库在阶段 0 骨架上贯通一个作品、一个媒体的合成只读 Source。输入为仓库内小型夹具，测试复制到临时 Source 后执行；原始媒体 SHA-256 为 `02a62d2dd57a1f140c251473fe869ec808f956bbb9391ea7c0bebea49ac96110`，扫描、读取和失败路径前后保持一致。所有数据库、WAL、运行描述符和媒体读取快照均位于临时 AppDirs，没有向 Source 写入文件。

纵向链路覆盖：

- 从空 AppDirs 启动并迁移独立 `control.db` / `catalog.db`；完成短时单次 Personal 配对、HttpOnly Session、CSRF、Host/Origin/Fetch Metadata 和 capability 检查；
- 创建 Library、只读 Source、规范 JSON RuleVersion 和 SourceRuleBinding；规则编译固定到 Job，tests-only 包变化不改变 `semantic_hash`；
- 持久 Scan Job 依次经过 `queued`、`running`、`publishing`、`completed`，完成 SourceWork/SourceMedia、完整 SHA-256 ContentBlob/FileLocation、Canonical Work/Media 与稳定 Binding；
- Catalog 先在 staging 写完整快照，再用短事务发布 `query_publication_id` 和 active pointer；失败扫描不替换旧 publication；
- REST snapshot 覆盖 Job、当前 publication、Work、Media；媒体内容覆盖 HEAD、完整 GET、单区间 `206`、`Content-Range`、稳定 ETag、`If-None-Match` 的 `304`、非法或多区间的 `416`，以及离线和读取时内容变化的稳定错误；
- `/ws/v1` 覆盖 `job.queued`、`job.progress`、`job.completed`、`job.failed`、`query.publication.published` 和 `session.revoked`，每连接 sequence 单调递增并按 capability 过滤；Session 吊销后现有 WS 以 `4401` 关闭，REST 与媒体访问同时失效；
- 实际关闭并重新启动完整 bootstrap 后，未吊销 Session、Job、publication 和媒体 snapshot 仍可读取；启动 reconciliation 同时覆盖“Catalog 已发布而 Job 未完成”和“Job 已完成但 publication 缺失”两个跨库状态。

2026-07-16 在 Windows 11 专业工作站版 10.0.28000、NTFS、Go 1.26.5 上，根级 `Check.ps1`、`go test ./...`、`go vet ./...`、`CGO_ENABLED=0 go test ./...` 和 `go build ./cmd/...` 均通过；生成的 OpenAPI 客户端与契约文件无漂移，`govulncheck` 1.6.0 对当前可达调用报告 `No vulnerabilities found.`。另以真正独立的 `galleryd.exe` 在随机 loopback 地址启动，独立 `galleryctl.exe health` 返回 `status=ok` 且两库均为 `ok`；Ctrl+C 后服务进入停止路径、删除所拥有的 runtime descriptor，并保留两库。`galleryctl` 只引用公开 `pkg/galleryapi` / `pkg/galleryversion`，边界测试拒绝后端 `internal` 依赖。

同版 Go 1.26.5 linux/amd64 工具链在 WSL2 Debian、DrvFS 上以 `CGO_ENABLED=1 go test -race ./...` 通过。Windows 本机 race 的 `WaitOnAddress` 环境限制仍与 EV-12 相同，不据此宣称 Windows race 已通过；WSL DrvFS 也不代表原生 ext4。

局限：这是单作品、单媒体、合成 Source 的正确性纵向切片，不代表真实规模或发行就绪。正式 FTS/排序分页、合法 Catalog/Overlay revision 元组、Overlay 重投影、Catalog 删除重建、多强杀点、磁盘满、GC/VACUUM、真实只读样本、浏览器 UI/PWA、CredentialStore、LAN 和目标平台发行仍属于 Architecture Proof 或后续门禁。当前媒体读取为正确性优先的 AppDirs 临时快照方案，不构成大文件、HDD、NAS 或并发吞吐结论。

## EV-14 Architecture Proof Slice（E1 + E4/E5）

2026-07-16，正式实现基于 Walking Skeleton 完成 Architecture Proof 正确性切片。规则闭环通过 `/api/v1/rules/validate`、`compile`、`dry-run`、`impact` 和 RuleVersion API 提供 Schema 感知默认值物化、精确 Canonical JSON、`package_hash`/`semantic_hash`/`rule_ir_hash`、参数特化、编译缓存、脱敏 Trace、Impact 与稳定字段错误；有限原语覆盖路径/文件、metadata selector/回退、作品与媒体身份、Creator/标签、媒体分类/排序、隐藏、封面、缺失/损坏 metadata 和数组条件，CEL Profile 对表达式/AST/输入/集合/cost/超时设限且不注册 I/O、网络、进程、时钟或随机能力。三种仓库内目录/metadata 形态由同一 Rule IR 执行，无 Provider 业务分支。

查询切片使用 SQLite FTS5 3.53.2：NFKC + Unicode case fold 的复核原文、CJK bigram、拉丁/数字/文件名 trigram、标题/Creator/标签/文件名检索、Library/Source 范围与 authorization scope、自然排序 v1、Canonical Work ID tie-break、签名 keyset cursor、查询/授权指纹和五分钟持久租约。每次响应固定服务端签发的 `query_publication_id`，其复合外键绑定合法 `(catalog_revision, overlay_projection_revision)`；WorkProjection、CreatorProjection、MediaProjection、关系、SortKey 和 FTS 只读该元组。测试证明多页无重复/遗漏，活动 publication 切换后旧 cursor 在租约内继续，查询、排序、授权或 publication 不匹配时拒绝，篡改与过期分别返回稳定 code，公共查询不含绝对路径。

| killpoint | 重启结果 |
| --- | --- |
| Job 已创建、尚未扫描 | 排队 Job 恢复执行并只发布一次 |
| 部分 staging | 候选不暴露，Job 以 `PROCESS_INTERRUPTED` failed，旧 publication 可读 |

## EV-21 阶段 1 收尾：拆分合并指纹强化与 bootstrap 就绪时序（E1）

## EV-26：阶段 3 Correctness 小型收尾——扫描默认策略、TOCTOU、内容确认状态拆分与措辞修正（E1）

2026-07-19，对 EV-25 扫描档案落地后遗留的一组正确性缺口做最小且完整的收尾。全程使用合成夹具与临时 AppDirs/Source，未访问任何此前真实媒体 Source 或性能测试目录。确认并修正以下问题：

1. **首次扫描默认策略**：`CreateScanWithProfile` 此前对未显式指定 `scanProfile` 的请求硬编码为 `incremental`，与规范要求的"按 Source 是否已有 publication 自动选择 index/incremental"不符。现改为新增 `catalog.Store.SourcePublished`：Source 尚无发布数据时自动选择 `index`，已有发布数据时自动选择 `incremental`；两种情况下持久化的 `request_json.scanProfile` 都是最终决定的具体值，不保存含糊的 `auto`。
2. **严格校验**：原 `normalizeScanProfile` 把任何非法字符串（拼写错误）静默归一化为 `incremental`；现改为 `validateScanProfile`，只接受空字符串与 `index`/`incremental`/`verify`，其余一律返回结构化 `VALIDATION_ERROR` 且不创建 Job。
3. **TOCTOU 复用窗口**：`incremental` 复用分支此前虽调用 `media.LocateSourceFile` 重新打开并 Stat 当前文件，却从未使用其返回的 size/mtime 与 discovery 观察比较，实质未做任何复核。现比较这次重新 Stat 得到的 size/mtime 与 discovery 观察，任一证据变化即放弃复用、改用刷新后的当前状态建立完整 Hash Job，不把旧 digest 与新文件属性混合发布。
4. **`last_confirmed_at`/`verifiedAt` 语义**：复用分支此前把确认时间写成 `time.Now()`，导致每次无变化的 incremental 重扫都错误推进确认时间；现保留 `LookupPriorObservation` 返回的既往确认时间，只有真正完成完整哈希（首次确认或 `verify` 强制重新哈希）才推进确认时间。
5. **`location_status` 与 `content_verification_state` 拆分**：`media_projections.location_status` 此前借用 `located_unverified` 值同时表达位置可用性与内容确认状态；新增 forward-only migration `00009_media_projection_verification.sql` 为 `media_projections` 增加独立的 `content_verification_state`、`verified_at` 列并回填历史数据，`location_status` 恢复为只表达 `present`/`offline`/`missing`/`inaccessible`。
6. **未确认媒体内容端点**：`GET/HEAD /media/{id}/content` 此前对 `located_unverified` 媒体复用 `LocationStatus<>"present"` 分支返回 `MEDIA_OFFLINE`，把"内容未确认"误报为"Source 离线"；新增 `CONTENT_NOT_VERIFIED` 错误码，内容端点检测到未确认状态时返回该码，不进入依赖 ContentBlob/ETag 的已验证读取路径；`available` 字段现只表达位置可用性，位置在线但内容未确认时仍为 `true`。
7. **Job API `scanProfile`**：`GET /jobs`、`GET /jobs/{id}` 的 DTO 转换此前完全未填充 `scanProfile` 字段；现对 `scan` 类型 Job 从持久化的 `request_json` 解析并返回，不依赖进程内临时状态。
8. **SourceWork 结构审查边界**：`CreateScanWithProfile` 新增规则——已有 publication 的 Source 显式请求 `index` 时复用既有 `CONFLICT` 码拒绝，不创建 Job、不改写 Binding/Catalog，避免 `index` 绕过阶段 1 依赖 ContentBlob digest 的 SourceWork 拆分/合并审查；首次无 publication 的 Source 仍可执行 `index`。

## EV-27：阶段 3 最终 Correctness 与 CI 收口——Recovery 竞争领取、Catalog 重建默认档案与 index observation 一致性（E1）

2026-07-20，针对 `d3670c9` 触发的 Windows、Ubuntu 两个 GitHub Actions Job 同时失败，以及另外两处独立发现的正确性缺口做严格限定范围的收口。全程使用仓库夹具、`t.TempDir()` 和临时 AppDirs，未访问任何此前真实媒体 Source。

**1. GitHub Actions 原始失败与根因**：`Go (windows-latest)`、`Go (ubuntu-latest)` 均失败在 `TestRealProcessKillpointMatrix/overlay_fact_preprojection`。Windows 表现为"已发布/恢复场景未切换 publication"；Ubuntu `-race` 表现为 Job 停在 `status=failed issueCode=PROCESS_INTERRUPTED failureRetryable=true`、`next_attempt_at` 为固定时钟之后一秒（永不到期）。修复前定向复现 `-count=100`：22/100 次失败，与两个平台报告的症状一致，证明是真实的、依赖 goroutine 调度顺序的竞态，不是环境噪声。

根因是**生产状态机缺陷**，不是纯测试编排问题：`internal/overlay/service.go` 的 `Reconcile` 对状态为 `queued` 且尚无 publication 的 `overlay_projection` Job 会直接调用 `s.Start(job.ID)`（回退为未受调度器保护的裸 goroutine），其自身注释却写着"无 publication 的 queued Job 由中央循环提交"——代码与注释相互矛盾。生产 bootstrap 中这条路径之所以不出错，是因为 `overlayService.SetDispatcher` 注入了真正的 `jobs.Scheduler`，其 `Submit` 内部的 `inflight` 集合天然去重同一 `jobID` 的并发提交；`internal/scanner` 的 `Reconcile` 从一开始就对 `StatusQueued` 直接 `continue`，把 queued Job 的领取完全留给中央 `recovery.Service`，两个服务的对称设计出现了不该有的分歧。killpoint 测试的 `runtimeSubmitter` 绕开了 Scheduler 的去重直接调用 `Execute`，且测试构造的 Overlay/Scanner Service 从未注入 Dispatcher，因而业务 `Reconcile` 的裸 goroutine 与测试模拟的中央 `ReconcileOnce` 提交会真实地并发抢夺同一个 Job：谁先到达 `job_attempts` 的 running 行，测试紧接着对"running"状态做的强制过期改写就可能命中这个刚被业务 Reconcile 悄悄启动的 Attempt，使其被判定为进程中断、转入 `failed` 且 `next_attempt_at` 落在未推进的固定时钟之后，永不到期。

**2. 修复**：`internal/overlay/service.go` 的 `Reconcile` 删除对 queued Job 的直接 `Start` 调用，改为与 `scanner.Reconcile` 对称——无 publication 的 queued Job 只由中央 `recovery.Service` 的 `ListRunnable`/`Submit` 领取。`internal/overlay/service_test.go` 中依赖旧行为（`Reconcile` 自行执行 queued Job）的 `TestReconcileQueuedAndPublicationControlGap` 同步改为断言 `Reconcile` 后 Job 仍为 `queued`，再显式调用 `Execute` 模拟中央循环领取，继续验证"queued Job 可恢复"这一必须保留的断言，不弱化任何持久退避语义。

## EV-28：阶段 3 Catalog publication 恢复模型收口——同一逻辑 Job 多 Attempt 的候选幂等重建（E1）

2026-07-20，修复 EV-27 记录为"独立发现且本轮不修复"的缺口：Scan Job 已通过 `BeginCandidate` 建立 staging candidate 后进程被强杀，同一逻辑 Job 的后续 Attempt 因 `catalog_revisions.job_id` 的扁平 `UNIQUE` 约束永久无法重新建立候选。全程使用仓库夹具、`t.TempDir()` 和临时 AppDirs，未访问任何真实媒体 Source。

**1. 复现**：`internal/recovery/killpoints_test.go` 中 `partial_staging`/`candidate_complete` 两个 killpoint 此前把 `verifyBackoffRetry` 固定为 `false`，只断言强杀后 Job 保持 `failed` 且不污染 active publication。修复前把两者临时改为 `true`（推进 Manual Clock 越过 `next_attempt_at` 并重新 `ReconcileOnce`，验证同一 Job ID 产生新 Attempt 并完成），`-count=5` 定向运行稳定复现：`INTERNAL_ERROR: constraint failed: UNIQUE constraint failed: catalog_revisions.job_id (2067)`，与 EV-27 记录的根因完全一致，证明修复前代码在该路径上会稳定失败而非偶发。

**2. Candidate 所有权模型裁决**：确认 `internal/scanner/service.go` 的 `Execute` 在每次 Attempt 开始时已经调用 `AbortCandidate(ctx, jobID)` 并附注释"同一逻辑 Job 的新 Attempt 复用 Job ID；先清理上次未发布候选"——这证明代码原有设计意图就是**模型 A：Candidate 归逻辑 Job 所有**（同一 `job_id` 任一时刻至多一个未发布 candidate，多个 Attempt 共同推进或从头重建），只是 `AbortCandidate` 只把旧行标记为 `aborted` 而不删除，`BeginCandidate` 又对同一 `job_id` 盲目 `INSERT`，两者组合才撞上历史遗留的扁平 `UNIQUE` 约束。模型 B（Candidate 归 Attempt 所有）会要求把 `job_id` 唯一约束换成 `(job_id, attempt_number)`，牵动 `overlay_projection_revisions`、`work_projections`、`media_projections`、`query_publications`、`content_blobs`、`file_locations`、`work_search`、`source_works`、`source_media`、`source_creators`、`creator_projections`、`work_creator_relations` 十余张表的外键/级联关系与 GC/租约保护逻辑，且与"同一 Job 最终最多一个 publication"的既有产品语义（`query_publications.job_id UNIQUE`，`00002_publications.sql` 起始即有）没有额外收益，因此按规范要求的"优先评估模型 A"确认可行后未采用模型 B。

**3. 修复**：`internal/catalog/store.go` 的 `BeginCandidate` 从盲目 `INSERT` 改为幂等恢复入口——同一事务内先查询该 `job_id` 既有的 `catalog_revisions` 行：不存在则按原路径创建；`status` 为 `staging` 或 `aborted`（上次 Attempt 未完成或已被清理）则调用新增的 `resetCandidateRevision` 先删除该行（外键级联清理 `overlay_projection_revisions`/`source_works`/`source_media`/`source_creators`/`content_blobs`/`file_locations`/`work_projections`/`media_projections`/`creator_projections`/`work_creator_relations`，并显式 `DELETE FROM work_search`——FTS5 虚表不受外键级联管理），再插入全新 revision，不复用可能部分写入的 staging 结果；`status` 为 `published` 说明该 Job 已经真正完成发布只是 control 侧尚未收到 `completed`（Saga gap），此时返回新增的 `CodeCatalogCandidatePublished`（`CATALOG_CANDIDATE_ALREADY_PUBLISHED`，同步进入 `AllCodes()`、`error.schema.json`、`openapi.yaml` 并由 `go generate` 刷新 `pkg/galleryapi`），不得再次构建或发布。`internal/scanner/service.go` 的 `Execute` 捕获该错误码，调用新增的 `recoverAlreadyPublished`：查询 `PublicationForJob`、标记 Overlay 快照已发布水位、通过 `RecoverCompleted` 把 control Job 收敛为 `completed`，不重复发出 `PublicationPublished` 依赖事件、不重复生成 publication。

**4. Schema 语义**：`catalog_revisions.job_id` 与 `query_publications.job_id` 的扁平 `UNIQUE` 约束（`00002_publications.sql`）均保持不变，未新增或修改任何 migration，catalog schema 仍为 `user_version=9`。前者的最终语义澄清为"同一 `job_id` 任一时刻至多一行 `catalog_revisions`"（由 `BeginCandidate` 的幂等重置维护，而不是"该 Job 一辈子只能拥有一行"）；后者继续保证"同一 `job_id` 最终最多一个 `query_publication`"。`internal/catalog/candidate_test.go`（新文件）新增 `TestQueryPublicationsJobIDIsUniqueAtDatabaseLevel` 直接在数据库层验证第二次插入同 `job_id` 的 `query_publications` 行会被 UNIQUE 约束拒绝。

**5. staging candidate 状态机**：`staging`（未 Validate/未 Publish，可安全重置）→ `published`（已进入 `query_publications`，不可重置，只能对账）→ `aborted`（Reconcile 对终态 Job 的收尾标记，同样可安全重置直至被 GC 回收）。`BeginCandidate` 的分支覆盖前三态，任何超出这三态的异常值走 `default` 分支返回既有的 `CodeCatalogCandidateInvalid`。

## EV-34：阶段 4 压力测试前 Correctness 补充修正——VerificationTarget 请求 publication 与 active publication 混用（E1）

2026-07-20，在 EV-33 声称"VerificationTarget 冻结身份执行阶段验证复核后确认与 EV-32 描述一致，本轮未发现新缺口"的基础上，针对同一批 VerificationTarget 代码重新独立核实一个此前所有轮次复核都未覆盖的维度——**媒体身份解析所依据的 publication 与 observation 查询所依据的 publication 是否始终是同一个**。核实结论：这是一个此前一直存在、且未被 EV-31/EV-32/EV-33 任何一轮发现的真实缺口，已修复并补充确定性回归测试。全程使用仓库夹具、`t.TempDir()` 和临时 AppDirs，未访问任何真实媒体 Source，未修改任何历史 migration。

**事实矩阵**：

| 问题 | 修复前是否真实存在 | 根因 | 采用方向 |
| --- | --- | --- | --- |
| HTTP 层媒体身份解析与 observation 读取来自不同 publication | 是 | `createMediaVerificationJob` 用 `GetMediaAt(ctx, requestedPub, mediaId)` 按客户端显式指定（可以是任意仍存在的历史 publication）解析媒体身份，随后调用的 `catalog.LookupPriorObservation` 却固定 `JOIN active_query_publication`，与 `requestedPub` 完全无关；当 `requestedPub` 是一个已经不是 active 的历史 publication 时，媒体身份（含 `ContentVerificationState`）来自该历史快照，observation 却来自当前 active，两者可能描述不同事实 | 显式 `queryPublicationId` 必须精确等于当前 active publication，否则拒绝为结构化 `CONFLICT`；媒体身份与 observation 统一从同一个已确认为 active 的 publication 解析 |
| `VerificationTarget` 未冻结请求所依据的 publication | 是 | `VerificationTarget` 结构体只有 `mediaId`/`sourceId`/`relativePath`/`observationFingerprint`，没有 publication 字段；EV-31 引入该类型、EV-32 补齐执行阶段对 MediaID/ObservationFingerprint 的比对时均未注意到这一遗漏 | 新增必填字段 `queryPublicationId`，由 `CreateVerificationScan` 校验其格式合法且等于当前 active publication |
| 执行阶段的 `verifyObservationUnchanged` 读取执行时刻的 active publication，而不是请求时冻结的 publication | 是 | 该函数调用同一个 `LookupPriorObservation`（固定 active-bound），与冻结目标完全无关；Job 创建后到真正 `Execute` 之间如果 active publication 发生变化，比较基准会静默改用新的 active，而不是请求方原本绑定的快照 | 新增 `catalog.Store.LookupObservationAt(ctx, publicationID, sourceID, relativePath)`，按显式 publication 查询；执行阶段改用该入口并传入冻结的 `target.QueryPublicationID` |
| 幂等键未绑定 publication，与注释声称不符 | 是 | 幂等键格式为 `verify-media:v2:<mediaId>:<sourceId>:<fingerprint>:<relativePath>`，函数文档注释却写"本次解析到的 `queryPublicationId`"参与派生，实际代码从未把 publication 编码进 key 字符串 | 幂等键升级为 `verify-media:v3:<queryPublicationId>:<mediaId>:<sourceId>:<relativePath>:<fingerprint>`，注释与实现保持一致 |
| 前置身份不匹配误用 retryable 的 `CONTENT_CHANGED_DURING_HASH` | 是 | `verifyObservationUnchanged` 检测到冻结指纹与执行时刻观察不一致时返回 `fault.New(fault.CodeContentChangedDuringHash, true, nil)`；这是 Job 真正开始读取内容之前的前置身份校验，不是完整 Hash 读取过程中的并发变化，对已经失效的请求快照重试没有意义 | 复用既有 `VERIFICATION_TARGET_MISMATCH`（不可重试），不新增错误码 |

以下逐项说明最终模型、修复前测试如何证明问题存在（均已用临时反转生产代码逻辑、保留新增字段签名以维持编译、单独运行新测试验证失败，再恢复为修复后代码）：

**1. 按需确认收敛为"当前 publication 操作"**（`internal/transport/httpapi/server.go`）：`createMediaVerificationJob` 先用 `GetMediaAt` 解析 `resolvedPub`/媒体，若 `requestedPub` 非空则额外取当前 `catalog.Current`，`resolvedPub.ID != current.ID` 时返回结构化 `CONFLICT`，不创建 Job、不修改 Binding/Catalog/Overlay/Source；省略 `queryPublicationId` 时 `resolvedPub` 天然等于 `Current()`，行为不变。产品语义收敛为"媒体按需确认只确认当前可见 Catalog 中的媒体，不重新确认历史 publication 描述的旧 observation"，不新增"历史媒体确认"能力。`internal/contract/api/openapi.yaml` 同步更新该端点与 `queryPublicationId` 参数的描述，`go generate` 刷新 `pkg/galleryapi`（仅文档字符串变化，无新增字段/错误码）。

**2. `VerificationTarget` 冻结 `queryPublicationId`**（`internal/scanner/service.go`）：结构体新增 `QueryPublicationID` 字段；`CreateVerificationScan` 在确认 Source 已发布后，取当前 `catalog.Current`，对每个 target 校验 `QueryPublicationID` 是合法 `qpub_` ID 且精确等于当前 active，否则分别返回 `VALIDATION_ERROR`（格式非法）或 `CONFLICT`（合法但非当前 active）；未发布 Source 上仍先返回既有的 `CONFLICT`，不因 `queryPublicationId` 校验改变这个更基础前置条件的错误码。新增 `scanner.ObservationFingerprint(size, mtimeNanos int64, state string) string` 作为指纹格式的唯一权威构造入口，替换掉此前 HTTP handler、Scanner 执行阶段和测试各自手写的 `fmt.Sprintf("%d:%d:%s", ...)`。

**3. **实现范围**：

1. 新增 forward-only control migration `00020_phase5_security.sql`，引入持久 Principal、LAN 本地账户、
   Role/capability、allow/deny Grant、API Token、Share、安全审计与登录限流状态；Personal owner 映射为
   持久 `personal-owner` Principal。旧 Session 因历史表中含可回显 CSRF 值而主动失效，不尝试不安全迁移；
   v19→v20 有数据升级测试确认 Canonical/Binding/Overlay 等用户事实不受影响。阶段 5 结构方向只登记为
   `COMPATIBILITY_BASELINE`，Argon2id、Session 和限流数值为 `PRE_FREEZE`。

4. API Token secret 只在创建响应展示一次，数据库只保存摘要和安全前缀；Token 的实际权限为当前
