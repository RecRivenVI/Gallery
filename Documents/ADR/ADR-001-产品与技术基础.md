# ADR-001 产品与技术基础

- 状态：接受
- 日期：2026-07-16
- 规范：[产品定义与不变量](../规范/01-产品定义与不变量.md)、[系统架构与模块边界](../规范/02-系统架构与模块边界.md)

## 问题

Gallery 需要在本地优先、跨平台、只读大媒体库、个人及可信 LAN、多客户端和小团队维护之间取得平衡。必须确定产品是否受现有系统约束，以及后端语言、服务形态、数据库和主 UI 的共同基础。

## 决策

Gallery 是独立产品，不把任何现有应用的数据库、配置、API 或行为作为兼容目标。后端使用 Go，采用 API-first 模块化单体和单主进程；默认数据库为 SQLite WAL，并分离 `control.db` 与 `catalog.db`；主业务 UI 为响应式 Web/PWA。长任务由 SQLite 持久 Job + 进程内有界调度完成，不默认引入微服务、外部队列或远程数据库。

## 理由

- 本地部署需要低运维、单机强事务、便携发行和明确数据目录；
- Go 对文件 I/O、取消、有界并发、单文件发行和跨目标构建较直接；
- SQLite 符合单写者批量发布、多读者查询和零服务依赖；
- 模块化单体保留清晰边界，避免把单机问题变成分布式一致性问题；
- Web/PWA 让普通浏览器、桌面和移动共享一套业务 UI 与 API；
- 发行原型表明 Go 在最小包体和启动上适合目标，见 [EV-11](../证据/验证记录.md#ev-11-后端发行技术对照e2e5)。

## 替代方案

- **Node.js/TypeScript**：Web 开发统一，但后端文件扫描、单文件发行和长期类型边界不是当前最优组合。
- **C#/.NET**：Windows 集成好，但便携体积、裁剪稳定性和跨平台发行成本更高。
- **Rust**：性能和安全优秀，但单人/小团队开发速度与生态集成成本更高。
- **PostgreSQL + 服务化组件**：适合多节点并发写，不符合默认单机零运维目标。
- **微服务/外部队列**：增加部署、版本、故障和一致性面，没有当前规模收益。
- **原生多端 UI**：需要维护多套体验；当前 API + Web 能覆盖主要场景。

## 影响

- 模块边界必须在单进程内清晰，不能用“单体”作为循环依赖理由；
- 任何客户端都通过正式 API 工作，数据库和扫描器不暴露给壳；
- SQLite 写入、备份和恢复成为核心工程能力；
- 特定平台集成经 adapter，不能污染领域核心；
- 未来若增加服务，可从已定义端口拆分，而不是预先分布式化。

## 重新审议条件

- 产品正式要求多主机共享任务队列或同一数据库的多实例并发写；
- 目标转为 Windows-only 并需要深度 COM/AD/企业集成；
- SQLite 在真实门禁中无法满足单机规模，且优化/分区仍失败；
- Web/PWA 在已验证高频核心场景中无法满足产品需求。
