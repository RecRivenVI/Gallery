# ADR-002 领域与数据所有权

- 状态：接受
- 日期：2026-07-16
- 规范：[领域模型与数据所有权](../规范/03-领域模型与数据所有权.md)

## 问题

扫描结果会随 Source、规则和文件变化而重建，但用户合并、覆盖、收藏和进度必须长期稳定。同时，逻辑媒体、字节内容和文件位置具有不同生命周期；如果压成一张“作品/文件”表，重扫、移动和去重会破坏用户状态。

## 决策

`catalog.db` 保存 SourceWork/SourceCreator/SourceMedia、ContentBlob/FileLocation 和可重建查询投影；`control.db` 保存 CanonicalWork/CanonicalCreator/CanonicalMedia、WorkBinding/CreatorBinding/MediaBinding、Overlay、账户和其他不可重建事实。Source-derived 实体通过稳定 Source 引用和持久 Binding 指向 Canonical 实体；`control.db` 不得保存 Catalog revision 内部 row ID。

CanonicalMedia 是 CanonicalWork 中的逻辑 occurrence；ContentBlob 是完整字节内容，v1 以算法版本 + 完整 SHA-256 确认；FileLocation 是可变位置。ContentBlob/FileLocation 随 Catalog revision 可重建，CanonicalMedia 的当前内容由 MediaBinding 和查询投影解析；固定内容版本在 control 中保存算法版本 + digest，而不是 Blob row ID。DerivedAsset 以 Blob 稳定引用、变换版本、参数和相关 Overlay 输入为键，其登记与缓存均可重建。

Provider 是轻量注册表；Source 与 Provider 多对多。CanonicalWork 可有多个 Origin；CanonicalWork 与 CanonicalCreator 通过带 role/ordinal 的关系多对多。

## 理由

- Catalog 可删后重扫，而用户事实不可恢复；两者必须有不同生命周期；
- 稳定 Binding 允许 Source 路径改变、来源消失和 Canonical 合并而不丢用户组织；
- 人工构造的采样碰撞证明快速签名不能确认内容；
- 复制、硬链接和同字节多 occurrence 证明 FileID、ContentBlob 和 CanonicalMedia 不能合并；
- 真实样本发现重复 ContentBlob 和多种 Origin/Creator 结构，见 [EV-03](../证据/验证记录.md#ev-03-媒体身份e1--e3e4)。

## 替代方案

- **所有状态放一个可重建 Catalog**：重扫会丢或覆盖用户事实。
- **增加第三个 Overlay 数据库**：增加跨库协调，没有独立生命周期收益；control 足以承载。
- **路径作为 CanonicalWork/CanonicalMedia 主键**：改名、移动和目录重排会制造新实体。
- **采样哈希作为 Blob 主键**：已被确定性碰撞否定。
- **FileID/inode 作为统一身份**：复制、跨卷和重挂载会改变，且不能表达逻辑 occurrence。
- **Provider 拥有平台业务对象和代码分支**：把规则差异固化到核心，破坏可配置边界。

## 影响

- 查询需要将 control Overlay 投影到 Catalog，但投影不是权威；
- 备份必须优先保护 control，Catalog/搜索/派生资源可重建；
- 重扫只能更新 Source-derived 行和 Binding 状态，不能删除 Canonical/Overlay；
- ContentBlob 去重只复用字节与派生计算，不自动合并 CanonicalWork/CanonicalMedia；
- 新 ContentBlob 的首次完整 SHA-256 是 publication 前置条件；超大文件或网络盘可以延后发布，但不能用快速指纹建立权威 Blob；
- DerivedAsset 数据库登记丢失后只能凭完整稳定 key 和受校验 manifest 接管缓存，不能按文件名猜测；
- Catalog 删除重建后必须以稳定 Source 引用和 Blob digest 恢复 Binding/固定内容，不能依赖旧 row ID；
- 最终数据库唯一约束必须等网络盘和文件身份门禁完成。

## 重新审议条件

实体边界属于产品不变量，仅在出现无法由 Binding/Origin/CanonicalMedia/ContentBlob/FileLocation 表达的真实生命周期时重开。哈希算法升级应增加 algorithm/version，不得退回路径或采样身份。

## 演进说明

- 2026-07-17：control.db 产品级备份/恢复落地（见 [验证记录 EV-19](../证据/验证记录.md)），作为本决策「control 最高备份优先级、Catalog 可重建」的实现基线，不改变实体边界或所有权矩阵。备份为 control.db 的 SQLite 一致性副本，附带 `manifestVersion=1` 的自描述 manifest（role、schemaVersion、migrationChecksum、checksum、安全范围）；恢复在下次启动、打开数据库前于 AppDirs 单写者锁下隔离迁移并原子替换，失败保留当前库，恢复后作废 Session/pairing 与非终态 Job。备份 manifest 版本与恢复应用模型的冻结阶段与重新审议门禁登记于 [v1 实施计划](../指南/01-v1实施计划.md) 的「暂定实装决策」表。此说明不新增独立 ADR。
- 2026-07-18：SourceWork 拆分/合并落地并执行阶段 1 Schema Freeze Gate（见 [验证记录 EV-20](../证据/验证记录.md)），阶段 1 结束。拆分/合并首先是 Source-derived 结构变化，不自动等同 CanonicalWork 拆分/合并；检测只用 ContentBlob digest 证据，无法安全自动处理时复用 Binding issue 阻塞 publication，人工决策以 control.db 的 pre-seed WorkBinding（继承写 inactive、排除写 manual_unbound）表达。撤回仅限尚未被扫描消费的 pre-seed 决策：清除本次 pre-seed Binding 并重开 issue；已消费时返回 `CONFLICT`，决策、issue 与 Binding 不部分修改，已生效结构变化需新的人工决策或未来补偿流程。决策与溯源持久化在 `source_structure_decisions`，Catalog 全量重建后可重放。此实现落在本决策「稳定 Binding、control 拥有人工决策、Catalog 可重建、不依赖 row ID、去重不合并 Canonical」的既有边界内，未改变实体边界或所有权矩阵，不新增独立 ADR。阶段 1 Schema Freeze Gate 把身份与唯一约束逐项分类登记在 control 迁移 `00016_schema_freeze_phase1` 的 `schema_freeze` 表：`(source_id, source_key) WHERE status='active'` active Binding 唯一、`(work_id, ordinal)` 唯一、CanonicalWork 仅以持久 ID 标识、同 Blob 多 occurrence、结构决策 fingerprint 唯一等冻结为兼容承诺（FROZEN），破坏前须新增或修订 ADR；orphan 阈值、external ID 冲突策略、RuleVersion 身份命名空间、WorkOrigin 模型、FileLocation 唯一约束等保持 COMPATIBILITY_BASELINE/PRE_FREEZE/DEFERRED，仍可经 forward-only migration 演进。本 ADR「最终数据库唯一约束必须等网络盘和文件身份门禁完成」的影响项据此细化为：Binding/CanonicalWork/CanonicalMedia 的核心身份约束已冻结，FileLocation 在 SMB/inode/无 FileID 下的唯一约束仍待文件身份门禁。
