# 验证记录

> 类型：验证证据。本文只保存仍支撑当前决策的聚合结果，不定义产品规范。原型源码和机器可读结果保留在 `Test-Bench/cleanroom-lab-real`；较早的技术对照保留在 `Test-Bench/cleanroom-lab`。

## 证据等级

| 等级 | 含义 |
| --- | --- |
| E1 合成单元/契约 | 可重复验证语义和边界，不代表真实规模或平台 |
| E2 合成性能 | 在记录机器和版本上的比较，可支持方向，不直接构成普遍 SLA |
| E3 真实只读样本 | 验证真实目录/metadata 形态；必须用前后 guard 证明零写入 |
| E4 目标平台运行 | 在目标 OS/文件系统实际执行核心或发行流程 |
| E5 交叉编译/推断 | 只证明可构建或提供估算，不代表运行支持 |

执行日期为 2026-07-16。结果来自本机固态存储、当时的 Go/SQLite/工具版本；正式实现必须在 CI 和目标硬件重跑。

## 性能证据环境与使用边界

EV-04/EV-05 的结果文件没有把完整主机信息写入机器可读输出；下表依据同日执行主机、结果 JSON、`go.mod` 和原型源码补齐可确认项。它是**参考机候选基线**，不是已冻结的官方 Reference Performance Gate；未控制冷缓存，因此不能据此声称冷启动 SLA。

| 项 | EV-04 Catalog | EV-05 搜索 |
| --- | --- | --- |
| CPU | Intel Core i7-14700K，20 核/28 线程 | 同左 |
| 内存 | 127.7 GiB | 同左 |
| 存储 | Crucial T500 2 TB NVMe SSD，NTFS，4 KiB allocation unit | 同左 |
| 操作系统 | Windows 11 专业工作站版，10.0.28000 | 同左 |
| 数据库/引擎版本 | SQLite 3.46.0；`modernc.org/sqlite` 1.34.4 | SQLite 3.46.0 / `modernc.org/sqlite` 1.34.4；Bleve 2.4.4，Scorch + BoltDB |
| 冷热缓存 | 未主动清空 OS 缓存；每个模型新建数据库，不能标记为严格 cold/warm | 在已建索引上串行重复查询，30 次 P50/P95；未主动清空 OS 缓存，视为 warm-ish |
| 并发 | 单基准进程；SQLite 单连接写入，读者探针用于确认旧 revision 可见 | 单基准进程、单查询串行；每次 limit 200 |
| 样本规模 | 1,000,000 WorkProjection；每个投影 2 条关系；1%/10%/50% 变化；每个组合 1 次观测 | FTS5 为 1,000,000/10,000,000 合成文档；Bleve 为 1,000,000；每类延迟 30 次 |

因此：EV-04 的 3–8 ms 是各组合的一次 publication 观测，不是 P95；EV-05 的 P95 是 warm-ish、串行、limit 200 的查询基线。正式门禁必须按 [测试与发布门禁](../指南/02-测试与发布门禁.md) 重新记录全部环境字段，并分别报告 Correctness、Reference Performance 和 Degradation 结果。EV-10/EV-11 的早期包体/启动数字缺少同等完整元数据，只能用于方向性对照。

## EV-01 真实媒体只读盘点（E3）

| 样本模型 | 目录 | 文件 | 作品估算 | metadata | 数据量 |
| --- | ---: | ---: | ---: | ---: | ---: |
| pixiv | 103,330 | 364,634 | 102,849 | 102,849 | 约 508.8 GiB |
| pawchive | 2,354 | 11,595 | 2,350 | 2,350 | 约 116.1 GiB |
| gank | 260 | 3,148 | 258 | 258 | 约 3.7 GiB |
| 合计 | 105,944 | 379,377 | 约 105,457 | 105,457 | 约 628.6 GiB |

抽样解析 4,258 份 metadata 均有效。主要文件包含 JPG、PNG、GIF、WebM、MP4、M4V、MOV、ZIP、RAR、PSD 和 CLIP。前后清单的 guard SHA-256、目录数、文件数和总字节数逐根一致，证明探测未改媒体根。

支撑：真实 Source 需要条件数组、非浏览器原生附件、非媒体作品和多种 metadata 形状；媒体根只读可验证。

局限：只覆盖当前三种目录模型和 Windows 可见卷，不代表网络盘或所有 Provider。

## EV-02 真实规则执行（E3）

对 5,608 个真实作品执行 10 类聚合规则：Pixiv 3,000、Pawchive 2,350、Gank 258。

| 发现 | 结果 | 决策影响 |
| --- | ---: | --- |
| Pixiv 固定 `cover` basename | 0/3,000 | 封面必须是候选评分，不是固定文件名 |
| Pawchive 附件数组 | 2,257/2,350 | 需要数组谓词、条件和缺失分支 |
| Pawchive cover | 1,906/2,350 | cover 是条件事实 |
| Gank postMedia 数组 | 258/258 | 需要数组排序、字段选择和隐藏策略 |
| Gank 简单 Creator 路径回退 | 0/258 命中 | 需要参数化 selector 和可解释 fallback |
| 无常见媒体扩展的作品 | Pawchive 136、Gank 5 | 不得自动判为损坏 |

支撑：[规则系统](../规范/05-规则系统.md) 的有限原语 + bounded CEL、Trace 和参数化 selector。

局限：字幕关联未在这些主要样本中出现，不能宣称已验证。

## EV-03 媒体身份（E1 + E3/E4）

Windows 和 WSL 夹具覆盖改名、同卷移动、复制、硬链接、symlink、同作品重复和跨 Work 重复。人工构造两个 192 KiB 文件：大小及首尾 64 KiB 完全相同，中间不同；快速指纹相同，完整 SHA-256 不同。

真实媒体各抽取 1,500 个完整内容哈希样本，共发现 83 组确认重复；未观察自然快速碰撞，但人工夹具已证明采样签名不能作为权威键。

支撑：CanonicalMedia、ContentBlob、FileLocation、DerivedAsset 分离；ContentBlob 由完整 SHA-256 确认；FileID/路径/采样仅作候选。

局限：跨卷实际移动、SMB 重挂载、inode 复用、稀疏文件和 junction/reparse 尚未运行。

## EV-04 Catalog 提交（E2）

同一百万 Source-derived/查询投影、关系、SourceMedia/ContentBlob/FileLocation 和 FTS 合成数据，对 1%/10%/50% 变化比较两种可恢复模型：

| 模型 | 变化 | 长构建 | 发布事务 | 发布前空间 | GC/VACUUM 后空间 |
| --- | ---: | ---: | ---: | ---: | ---: |
| generation-delta | 1% | 104 ms | 960 ms | 468.0 MiB | 213.6 MiB |
| generation-delta | 10% | 819 ms | 4.701 s | 487.5 MiB | 218.0 MiB |
| generation-delta | 50% | 4.017 s | 20.774 s | 577.7 MiB | 237.0 MiB |
| staging-snapshot | 1% | 40.730 s | 4 ms | 610.9 MiB | 214.5 MiB |
| staging-snapshot | 10% | 40.526 s | 3 ms | 610.7 MiB | 213.9 MiB |
| staging-snapshot | 50% | 40.431 s | 8 ms | 604.1 MiB | 211.1 MiB |

两种模型在构建和回滚探针中都保持旧快照可读，Catalog 与 FTS 同一 Catalog revision。staging 的单次 publication 观测为 3–8 ms 且未随变化比例增长，因此支撑默认 staging snapshot + publish pointer 的方向；它不等于已经通过正式 P95 门禁。

局限：本机 SSD；staging 的完整构建、约 604–611 MiB 峰值、约 14–15 秒 GC 和约 6.5 秒 VACUUM 需要在 HDD 重测；该原型未覆盖独立 Overlay projection revision，其配对发布和重投影性能仍需 Architecture Proof 验证。

## EV-05 搜索（E2）

实现 CJK bigram、拉丁/文件名 trigram 和规范化原文复核。Bleve 对照使用默认字段映射、CJK analyzer、MatchQuery、Scorch/BoltDB；没有为 filename infix 设计专用 n-gram analyzer 或 query：

| 指标 | FTS5 100 万 | FTS5 1000 万 | Bleve 100 万 |
| --- | ---: | ---: | ---: |
| 构建时间 | 28.704 s | 333.083 s | 21.087 s |
| 索引/文档体积 | 285.6 MiB | 2.789 GiB | 209.1 MiB |
| 选择性 CJK 分页 P95 | 0.725 ms | 1.838 ms | 39.033 ms，且召回过宽 |
| 选择性文件名分页 P95 | 1.665 ms | 2.759 ms | 当前 analyzer/MatchQuery 下宽查询 0 命中 |
| 宽 CJK 前 200 条 P95 | 1.044 ms | 1.848 ms | 17.981 ms |

双字/多字中日文、英文、全角半角、大小写、数字、假名和文件名中缀的候选召回与复核准确率均为 1.0。千万样本宽查询若把全部候选搬到 Go 需约 9.4–14.8 秒，证明必须分页和尽早复核。

支撑：v1 FTS5 限定能力。当前 Bleve 原型在所采用的 analyzer 与 query 配置下未通过 filename infix 门禁，并且会引入独立索引与 Catalog revision 的协调成本，因此不作为 v1 默认引擎；该结果不能扩大为“Bleve 本身不支持文件名中缀搜索”。

局限：合成短文本；单字符 CJK、拼音、模糊、正式排名/高亮和 Bleve 千万级未验证。

## EV-06 排序协议（E1）

黄金样本验证 `file2 < file10`、全角/半角和大小写等价、emoji 数字自然序、nulls last、相同键下独立 ID tie-break。Windows、Linux、macOS 目标使用同一 Go 生成器；Linux/macOS 此处是同源构建证据，不是完整目标平台 UI 对拍。

支撑：排序协议 v1 和 `ORDER BY sort_key, work_id`。

局限：完整 grapheme、null 降序和正式 `sort_reading` 语料仍需扩充。

## EV-07 规则与 CEL（E1 + E3）

自建原语、CEL 和 Starlark 均能表达最小规则目标；可分析性、安全性和 UI 生成能力依次下降。CEL 原型编译并限成本执行 4 个条件/集合谓词，拒绝不存在的 host 函数；真实规则结果见 EV-02。

支撑：规范 JSON + JSON Schema + 有限原语 + 受限 CEL，配置内不执行 Starlark/JavaScript。

局限：未实现插件/WASM；这两项已延后，不阻塞 v1。

## EV-08 跨库、游标和 Personal 契约（E1）

原型验证：

- Catalog 已发布而 Job 仍 publishing，可修正 completed；
- Job completed 而无 publication，进入 needs_repair；
- 游标对 query、sort version、内部 snapshot revision、auth scope、GC 和 HMAC 篡改产生预期分支；
- Personal 匿名非管理员、首次配对、配对码复用拒绝、Host/Origin 拒绝、CSRF、capability 交集、多标签共享和 Session/WS 吊销均通过。

支撑：跨库 Saga、快照分页和 Personal 浏览器配对。

局限：原型验证了游标绑定内部 snapshot revision，尚未单独验证公共 `query_publication_id` 对合法 Catalog/Overlay revision 元组的封装；这属于查询 Schema 冻结门禁。Sec-Fetch-Site 浏览器 E2E、各浏览器 Cookie 细节和真实 CredentialStore 也尚未完成。

## EV-09 路径和跨平台核心（E1 + E4/E5）

常见 traversal、绝对/盘符/UNC、Windows 保留名、尾点/尾空格、NUL 和 symlink 逃逸共 14/14 按预期拒绝。相同核心探针在 Windows 11/NTFS 和 WSL Debian/DrvFS 实际运行；Linux amd64 与 macOS amd64 完成交叉编译。

Windows/DrvFS 验证大小写行为、NFC/NFD 名称并存、hardlink/symlink、长路径夹具、平台 FileIdentity 和工具发现。WSL 使用 DrvFS，不能代替原生 ext4；macOS 只可构建；Docker daemon 未运行。

支撑：跨平台核心 + 适配层、路径不是身份、Watcher 只作提示。

局限：ext4、APFS/FSEvents/Keychain、junction/reparse、SMB/NAS、Docker 和 HDD 均待门禁。

## EV-10 后端独立与桌面壳（E1 + E4/E5）

独立 `galleryd` 模拟器在 Windows 和 WSL Linux 验证 loopback 动态端口、原子 descriptor、健康检查、单实例锁、被杀后重启和重连；实现不引用 Wails。

Wails 2.13.0 Windows 最小壳构建约 28.4 秒，产物约 11.0 MiB，启动后 3 秒工作集约 27.4 MiB。它只验证壳体积和启动，不含真实后端、托盘、自启、目录选择、更新或签名。

支撑：后端独立、壳可替换；Wails 仅有条件接受。

## EV-11 后端发行技术对照（E2/E5）

早期最小发行原型中，Go 静态二进制约 6 MiB、启动约 154 ms；含 SQLite/Bleve 的组合约 6–16.6 MiB，Linux 交叉编译产物约 6.25 MB。ASP.NET self-contained 未裁剪约 98.8 MiB、启动约 686 ms；激进裁剪出现运行时 `TypeLoadException`。

支撑：在本地优先、单文件、跨平台和小团队维护目标下选择 Go。

局限：最小原型不代表完整产品包体；Go idle 内存读数受环境干扰，未作为决策依据；Docker 镜像大小只是静态二进制 + scratch 的推断。

## EV-12 阶段 0 正式契约骨架（E1 + E4/E5）

正式仓库使用 Go 1.26.5 和 `modernc.org/sqlite` 1.53.0 建立第一条可运行工程基础。在 Windows 10.0.28000/NTFS 上，`CGO_ENABLED=0 go test ./...`、`go vet ./...` 和 `go build ./cmd/...` 通过；真实 `galleryd` 进程以动态 loopback 端口启动，创建独立 WAL `control.db` / `catalog.db`、原子 runtime descriptor，并由生成的 `galleryctl` OpenAPI client 获得两库 `ok` 后优雅停止。合成 Source 的 SHA-256 在启动前后相同；AppDirs/Source 重叠在目录创建和数据库初始化前失败。

数据与契约测试覆盖两库 migration 版本/历史 checksum、防部分 migration、control 独立备份与完整性检查、带实体前缀 UUIDv7、完整 SHA-256 Blob 引用、HMAC 游标篡改/过期、错误/WS/规则 JSON Schema、匿名非管理员和匿名 WS 拒绝。`govulncheck` 1.6.0 对当前可达调用未发现漏洞。

同版工具链在 WSL Debian/DrvFS 上执行 `go test -race ./...` 通过。Windows 本机 race 二进制在测试代码运行前统一以 `0xc0000139` 退出；检查确认该系统的 `kernel32.dll` 不导出 Go race runtime 静态导入的 `WaitOnAddress`，而 `kernelbase.dll` 仍导出它。因此这是当前 Windows build 与 Go race runtime 的环境兼容限制，不记为代码竞态通过；Linux CI 保留 race 门禁。

局限：该证据仅覆盖阶段 0 合成正确性和当前 Windows/WSL 运行，不包含 Walking Skeleton 的配对、扫描、完整哈希 publication、媒体 Range 或 Job WebSocket；WSL DrvFS 仍不代表原生 ext4，也不提升任何平台支持级别。

## 仍需重测的关键门禁

1. HDD 上 staging 构建、哈希、GC/VACUUM 和空间峰值。
2. Linux 原生 ext4/inotify、macOS APFS/FSEvents/Keychain 和 Docker 非 root 只读挂载。
3. SMB/NAS/UNC 断线、重挂载和文件身份变化。
4. Windows junction/reparse 与句柄式 TOCTOU 复核。
5. 正式搜索排名、高亮、宽查询 total、emoji/grapheme 和游标租约。
6. Wails/Tauri 的真实后端、更新、签名、公证和系统集成对照。

以上门禁的发布约束见 [测试与发布门禁](../指南/02-测试与发布门禁.md)。
