# 规则系统

> 类型：规范。规则格式、编译、执行、解释和影响分析以本文为唯一权威来源。

## 目标

规则系统负责把不同 Source 的目录、metadata 和媒体形态解释为统一的 Source-derived 事实。它必须同时满足：可配置、可复用、可静态分析、可生成表单、可解释、可测试、可版本化和默认安全。

规则不是平台代码替代品的别名。Provider 名称不能触发业务 `switch`；相同 RuleVersion 可被多个 Source 以不同参数复用。

## 规则实体

| 实体 | 职责 |
| --- | --- |
| RuleSet | 稳定规则家族 ID、所有者、参数 Schema 和发行通道 |
| RuleVersion | 以 `semantic_hash` 标识的不可变运行语义；分发包完整性另由 `package_hash` 标识 |
| SourceRuleBinding | Source 绑定 RuleVersion、参数、Provider 范围和优先级；与 WorkBinding/CreatorBinding/MediaBinding 无关 |
| RuleParameters | 经 Schema 校验的绑定值，不复制 RuleVersion |
| RuleIR | 由 compiler/profile version 生成的不可变执行计划和 hash |
| RuleTrace | 结构化步骤、JSON Pointer、候选、选择结果和原因码 |
| RuleImpact | 受影响字段、Source、重扫/重建搜索/重建派生资源要求 |

## 唯一事实格式

运行时唯一事实源是**规范 JSON 规则包**。JSON Schema 负责结构、约束、默认值、编辑器元数据和参数表单；有限原语负责常见业务；CEL 负责受控条件与集合谓词。

YAML/TOML 可以作为显式导入格式，但导入后必须转换为规范 JSON，不能成为并行配置路径。CUE 可在开发期生成或校验，不进入运行时。发行包只是封装，不改变内部模型。

RuleVersion 至少包含：

```text
rule_set_id
version
schema_version
normalization_algorithm_version
compiler_requirement
cel_profile_version
parameter_schema
provider_namespaces
primitives
cel_expressions
tests
extensions
package_hash
semantic_hash
```

未知扩展字段必须位于命名空间内，并在读取、表单编辑和原子保存时无损保留；严格 Schema 下的非扩展未知字段是阻塞错误，不能静默删除。

## 规范化、内容寻址与编译流程

```text
parse input
  → import conversion（YAML/TOML 等显式导入）
  → materialize JSON Schema defaults
  → normalize Unicode / number / object representation
  → canonicalize JSON
  → structural JSON Schema validation
  → semantic and reference validation
  → calculate package_hash
  → project execution semantics and calculate semantic_hash
  → parameter binding validation
  → primitive/CEL type and limit checks
  → static impact analysis
  → immutable Rule IR + hash
  → golden/sample tests
```

YAML/TOML 只是输入转换器。导入器必须拒绝不能无损映射到 JSON 的值和含糊隐式类型；导入结果进入同一默认值实例化与规范化流程，不能保留并行语义。相同语义的 JSON、YAML 和 TOML 输入必须生成逐字节相同的规范 JSON。

### Schema 感知的 Gallery Canonical JSON v1

- JSON Schema 的 `default` 只定义默认值；Gallery 的规范化阶段按 Schema 递归实例化默认值。缺省字段与显式填写同一默认值生成相同 RuleVersion；无默认值的缺省与显式 `null` 保持不同。
- 字符串默认 `preserve`。只有 Schema 明确声明规范化策略的字段才可执行 NFC、标识符规范化等变换；regex、glob、路径字面量、JSON Pointer、metadata 键和 external ID 必须逐 code point 保留，不做 Unicode 归一化、case-fold 或路径规范化。搜索 NFKC 与运行时 PathCanonicalizer 也不得反向改写规则包。
- 对象属性名和未知扩展命名空间默认保持原样，仅按原始 UTF-8 字节排序；只有 Schema 明确允许规范化的动态键才可变换，并在变换后冲突时阻塞。数组保持原顺序；输出 UTF-8、无 BOM、无无意义空白，并使用唯一字符串转义。
- JSON 数字必须从词法形式解析为“符号 + 任意精度整数 coefficient + 十进制 exponent”，以精确十进制算法规范化；禁止经 `float64`、YAML 浮点值或平台 locale 中转。`-0` 归一为 `0`，`1`、`1.0` 和 `1e0` 生成相同表示；位数和 exponent 有资源上限，Schema 范围比较也使用精确值。若目标 CEL 类型最终为 double，转换只发生在校验后的编译阶段，不能影响任何内容 hash。
- `normalization_algorithm_version` 必须版本化；算法变化不得静默重算既有 RuleVersion。

### package_hash

`package_hash` 标识收到的完整规则包，用于分发、签名、缓存和审计：

```text
SHA-256("gallery-rule-package\0canonical-json-v1\0" + canonical_package_without_hashes_and_signatures)
```

- 除 `package_hash`、`semantic_hash`、外层签名和传输封装外，parameter schema、tests、说明/展示元数据和全部 extensions 都参与；
- 修改 tests 或说明会产生新的 `package_hash`。已发布的包记录不可被同名覆盖，但这不自动产生新的运行语义。

### semantic_hash

`semantic_hash` 是 RuleVersion 的运行语义身份，输入是从规范包按 Schema 投影出的 canonical semantic model，至少包含 `rule_set_id`、Schema/规范化版本、parameter schema、provider namespace、primitives、CEL、声明的 compiler/Profile 要求和标记为 semantic/required 的 extension：

```text
SHA-256("gallery-rule-semantic\0v1\0" + canonical_semantic_model)
```

tests、说明、图标、发行说明和仅用于编辑器/分发的 optional extension 不参与 `semantic_hash`。因此只修改 tests 会改变 `package_hash`，但不会改变 RuleVersion 身份或触发重扫；若测试修订同时揭示规则行为需要改变，必须修改相应语义字段，从而产生新的 `semantic_hash`。未知 required/semantic extension 阻塞编译；未知 optional 非语义 extension 无损保留，仅参与 `package_hash`。

extension 的分类由每个 namespace 值声明：分类 extension 是对象 `{required: bool, semantic: bool, version?: string, payload?: any}`。四种组合的身份语义为：required+semantic 参与 `package_hash` 与 `semantic_hash`、未受支持时阻止编译；optional+semantic 参与两者、未受支持时显式拒绝而非静默忽略；required+nonsemantic 只参与 `package_hash`、未受支持时阻止编译；optional+nonsemantic 只参与 `package_hash`、容忍未知 namespace。required 或 semantic 者的 namespace 与 version 必须落在编译器支持的注册表内。extension 之间的顺序不影响任何 hash；相同 semantic payload 经规范化后得到相同 hash。未按上述结构声明（缺少 `semantic` 字段）的遗留 extension 一律按 optional+nonsemantic 处理，因此既不改变既有 RuleVersion 的 `semantic_hash`，也不会因未知 namespace 阻止编译。支持的 extension namespace 注册表是 pre-freeze 集合，冻结门禁与暂定实装决策见 [v1 实施计划](../指南/01-v1实施计划.md)。未受支持的 required/semantic extension 以 `RULE_SCHEMA_INVALID` 及对应 `/extensions/<namespace>` 字段拒绝。

### rule_ir_hash

`rule_ir_hash` 标识实际可执行计划，至少覆盖 `semantic_hash`、实际 compiler version、CEL Profile implementation version、primitive registry version、规范化算法版本，以及会使执行计划特化的规范化 RuleParameters hash。相同 RuleVersion 可因编译器/Profile 升级得到不同 `rule_ir_hash`，但 `semantic_hash` 不变。

运行中的扫描只使用任务开始时冻结的 SourceRuleBinding、RuleVersion、RuleParameters 和 Rule IR，不热切换半次扫描。

## 有限原语

优先用可解释原语表达以下能力：

- 路径层级、glob、受控正则和命名 capture；
- 安全 selector、字段回退链和类型转换；
- 时间解析、IANA 时区和原始文本保留；
- SourceCreator/SourceWork/SourceMedia 稳定键生成；
- 媒体分类、排序、隐藏、字幕关联和非媒体文件识别；
- 封面候选、评分、显式选择和背景继承；
- metadata 字段映射、标签规范化和 Provider Origin；
- 条件分支、缺失处理和结构化 warning/error。

跨记录 Blob 去重、文件 I/O、网络访问、压缩包解析和全局聚合不属于规则表达式；它们由核心服务或未来版本化插件承担。

## Gallery CEL Profile v1

CEL 只用于只读布尔条件、集合谓词和简单 scalar 选择，不能成为任意脚本层。

| 项 | Profile v1 |
| --- | --- |
| 变量 | `source`、`path`、`file`、`metadata`、`candidate`、`params`；均只读 |
| 类型 | JSON null/bool/int/uint/double/string/list/map；时间由原语解析为 timestamp |
| 函数 | 比较、布尔、`size`、contains/startsWith/endsWith、RE2 `matches`；自定义函数必须版本化 |
| 用途 | boolean 条件、`exists/all` 集合谓词、简单 scalar 选择 |
| null | 安全 selector 对缺失返回 null；非法直接访问产生结构化错误，不自动变 false |
| 规范化 | 显式 NFKC/case-fold 原语，不使用系统 locale |
| 表达式长度 | 最大 4,096 bytes |
| AST | 最大 256 nodes |
| 数组 | 最大 10,000 元素 |
| 输入 JSON | 最大 4 MiB |
| RE2 | 最大 512 characters |
| cost | 最大 10,000 |
| 单次执行 | 最大 10 ms；实现还必须支持上下文取消 |
| 禁止 | 文件、网络、进程、时钟、随机、反射、递归、任意 host 函数 |

若同一规则需大量 CEL 才能表达常规字段读取，说明缺少合适原语，应扩展版本化原语而不是放宽 Profile。

## 错误和解释

- 可选字段失败：产生 Trace warning，允许确定性回退；
- 必需身份字段失败：产生 `RULE_EVAL_ERROR`，隔离作品并按 Source publication policy 决定是否阻塞；
- 资源上限：产生明确 limit code，不按 false 继续；
- Explain 默认只输出 expression/primitive ID、输入 JSON Pointer、候选数、结果、cost、耗时和原因码；
- metadata 实际值仅在具备 `rules.debug` capability 的受控 Dry Run 中显示，并遵守脱敏策略。

## Impact 与版本变化

RuleImpact 必须静态区分：

| 变化 | 最低动作 |
| --- | --- |
| 名称、说明、图标等展示元数据 | 无需重扫；重建展示投影或刷新客户端 |
| 有效标题/标签/封面解析 | 重投影；必要时重建搜索/派生资源 |
| SourceWork/SourceMedia 身份或目录结构 | 对受影响 Source 全量重扫，并触发 Binding 审查 |
| 媒体可见性、排序或分类 | 重扫或重建相关 Source 投影 |
| 声明的 CEL Profile、原语语义或语义规范化算法 | 生成新 `semantic_hash`；实际 compiler/Profile 实现变化只生成新 `rule_ir_hash`，按影响字段决定范围 |

UI 不得凭字段名猜测重启或重扫要求，必须消费 RuleImpact。

## 编辑和保存闭环

- Schema 驱动表单、默认值、帮助、分组和高级字段；默认值由 Gallery 规范化阶段实例化，不依赖校验器副作用；
- 表单与高级 JSON 编辑器往返同一规范对象；
- 保存前使用与后端启动相同的 Schema/业务校验器；
- 保存采用临时文件、回读校验、原子替换和最近有效备份；
- 外部修改通过 revision/token 检测，禁止覆盖未知新版本；
- 提供按字段撤销、整包回滚、Dry Run、Explain、Impact 和版本 diff。

## 必需测试

每个 RuleVersion 包必须附带最小样本和断言，覆盖成功、缺失、类型错误、条件数组、路径边界、身份稳定、媒体顺序、封面、未知扩展和限额。还必须证明缺省/显式默认、对象键顺序和精确等值数字产生相同 `semantic_hash`；Schema 标记为 NFC 的普通文本按声明收敛，而 regex、glob、路径、JSON Pointer、metadata 键和 external ID 保持逐 code point 不变；JSON/YAML/TOML 等价导入不得因浮点中转漂移。tests-only 修改只能改变 `package_hash`，compiler/Profile 实现变化只按上述边界改变 `rule_ir_hash`。内置规则还必须在脱敏真实样本上验证，并确保新增字段在 JSON Schema、默认包、编译器、编辑器和文档中同步出现。

规则系统的已验证真实需求和局限见 [验证记录](../证据/验证记录.md)。对应决策见 [ADR-004](../ADR/ADR-004-规则系统.md)。
