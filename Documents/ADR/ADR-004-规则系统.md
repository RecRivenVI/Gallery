# ADR-004 规则系统

- 状态：接受
- 日期：2026-07-16
- 规范：[规则系统](../规范/05-规则系统.md)

## 问题

不同 Source 的目录、metadata、媒体顺序、封面和条件分支差异很大。规则既要覆盖真实数据，又要能生成 UI、静态分析影响、解释结果、限制资源并安全保存。单一 DSL 或任意脚本都难以同时满足这些要求。

## 决策

规则采用分层模型：规范 JSON 规则包作为唯一事实源；JSON Schema 管结构和参数 UI；有限原语处理 selector、回退、时间、路径、媒体和封面等常见业务；Gallery CEL Profile v1 只处理受控布尔条件、集合谓词和简单 scalar 选择；编译为不可变 Rule IR，并产生 RuleTrace 和 RuleImpact。

所有 JSON 或显式 YAML/TOML 导入先实例化 Schema default，再经版本化、Schema 感知的字符串/数字/对象规范化。regex、glob、路径、JSON Pointer、metadata 键和 external ID 保持原值；数字以任意精度十进制处理，不经 `float64`。

规则使用三类 hash：`package_hash` 覆盖完整分发包并随 tests/说明变化；`semantic_hash` 只覆盖运行语义，是 RuleVersion 身份，tests-only 修改不改变它；`rule_ir_hash` 还覆盖实际 compiler/Profile/primitive 实现和计划特化参数。这样既能审计包完整性，又不会把测试维护误判为业务语义变更。

配置内禁止任意 JavaScript、Starlark 或系统调用。插件/WASM 延后，只有真实规则无法覆盖时再设计版本化安全边界。

## 理由

- 有限原语可生成表单、做完整静态分析并天然 Explain；
- CEL 提供类型检查和受控表达力，适合真实样本中的条件数组；
- 任意脚本的安全、可分析性、兼容和 UI 生成成本不可接受；
- 5,608 个真实作品证明大部分需求是 selector/数组/封面/缺失处理，见 [EV-02](../证据/验证记录.md#ev-02-真实规则执行e3)；
- CEL 限额与非法函数原型已验证，见 [EV-07](../证据/验证记录.md#ev-07-规则与-cele1--e3)。

## 替代方案

- **纯 JSON Schema**：能校验结构，不能表达运行条件和集合谓词。
- **纯 CEL**：UI 生成、Impact 和细粒度 Trace 变差，常规字段读取会变成脚本堆积。
- **Starlark/JavaScript**：表达力高，但安全、终止、静态分析和长期兼容不足。
- **CUE/Jsonnet/Rego 作为运行时**：各自适合生成或策略场景，但不匹配本产品的表单、Explain 和媒体规则组合。
- **代码插件优先**：部署、权限和 ABI 成本过早，易恢复平台硬编码。

## 影响

- 需要维护 Schema、编译器、Profile、Rule IR 和原语版本；
- 需要维护版本化 Canonical JSON 算法，以及 package/semantic/Rule IR 三类 hash 的独立兼容边界；
- 表单和高级编辑器必须无损往返同一 JSON；
- SourceRuleBinding 参数与 RuleVersion 分离，多个 Source 可安全复用；
- 规则变化必须通过 RuleImpact 决定重扫/重投影，而不是 UI 猜测；
- 内置 RuleVersion 必须携带样本和黄金断言。

## 重新审议条件

至少两个真实 Source 的核心规则无法由有限原语 + bounded CEL 表达，且新增原语会明显破坏通用性时，评估版本化外部进程或 WASM。即使引入插件，规范 JSON、Schema、Rule IR、Trace 和 Impact 层仍保留。
