# ADR-005 查询协议

- 状态：接受（限定 v1 能力）
- 日期：2026-07-16
- 规范：[查询、搜索与排序](../规范/06-查询-搜索与排序.md)

## 问题

Gallery 需要在中日英混合文本、文件名中缀、大规模 Catalog、多客户端和扫描期间持续浏览之间给出稳定语义。搜索引擎、排序键和分页如果分别决定，会出现召回缺失、跨平台顺序漂移和翻页重复/遗漏。

## 决策

v1 使用与 Catalog 同库、与查询投影同 revision 元组的 SQLite FTS5：CJK bigram、拉丁/文件名 trigram 生成候选，NFKC + Unicode case fold 的原文做连续复核，结构化关系做精确过滤。

排序协议 v1 使用 NFKC、Unicode case fold、自然数字、稳定 Unicode 回退和 nulls last；服务器预计算排序键，SQL 以独立 `canonical_work_id` 打破平局。

有序列表承诺 publication 快照分页。服务端为合法 `(catalog_revision, overlay_projection_revision)` 创建 `query_publication_id`；签名游标绑定查询指纹、排序协议、该 publication ID、授权范围和末尾 `(sort_key, canonical_work_id)`。客户端不能组合裸 revision；不匹配或 publication 不可用返回 `CURSOR_EXPIRED`，篡改返回 `CURSOR_INVALID`。

Overlay 是否进入快照由当前查询的过滤、搜索、排序、可见性或集合依赖决定，不按字段永久分类。进入依赖集合的值来自 publication 快照；只用于展示的同一字段可按响应时当前值附加，但不得改变当前页集合和顺序。Overlay 事实同步写 control、异步发布投影；需要列表读己之写时使用 Overlay fact version 屏障。

阶段 4 Correctness 收尾（2026-07-20）把上述"不按字段永久分类"落实为服务端 planner 代码：`query.Service.Search` 按本次实际请求（用到的过滤字段、是否带搜索词）生成 `dependencySet`，静态的 `overlay.OverlayFieldCapabilities` 只登记字段能力，不再充当任何具体查询的最终依赖集合；Favorite/Progress 因为新增了 `overlay.favorite`/`overlay.progress` 过滤字段，从纯 `live` 升级为"写入触发快照屏障 + 额外提供 live 展示"。Ranking 协议同步从 v1（只有标题、0–3 单一 tier）升级为 v2（标题/Creator/Tag/文件名字段级 `match_class × field_priority` 组合元组，tier 范围扩大为 0–33），`rank_protocol_version` 递增使 v1 游标随之失效；高亮响应从 `titleHighlights` 改为通用 `matches: [{field, value, spans}]`。

## 理由

- 原生 FTS trigram 对双字 CJK 曾返回零结果；自建 bigram + 原文复核解决该缺陷；
- 100 万/1000 万合成证据中核心召回和准确率为 1.0，warm-ish 串行、limit 200 的千万分页 P95 <3 ms；该结果是参考基线，不是所有硬件 SLA；
- 当前 Bleve 原型在默认字段映射、CJK analyzer 与 MatchQuery 配置下未通过 filename infix 门禁，并且会引入独立索引与 Catalog revision 的协调成本，因此不作为 v1 默认引擎；这不等于 Bleve 本身不支持文件名中缀搜索；
- 自定义排序协议避免系统 locale 和客户端语言差异；
- 快照游标使扫描发布和权限变化不造成当前分页重复/遗漏；
- 证据见 [EV-05](../证据/验证记录.md#ev-05-搜索e2)、[EV-06](../证据/验证记录.md#ev-06-排序协议e1) 和 [EV-08](../证据/验证记录.md#ev-08-跨库游标和-personal-契约e1)。

## 替代方案

- **FTS5 原生 unicode61/trigram**：无法满足双字 CJK 或短查询。
- **Bleve 默认索引**：语言分析能力较强，但当前原型配置未通过 filename infix 门禁，并增加双存储 revision 协调。
- **Meilisearch/Typesense 等外部服务**：部署与一致性成本不符合本地 v1。
- **系统 ICU/locale 排序**：不同 OS、版本和客户端会漂移。
- **OFFSET 分页**：深页成本高，且新 revision 发布时容易重复/遗漏。
- **弱一致游标**：实现简单，但不能兑现可解释的浏览体验。

## 影响

- v1 明确不承诺无过滤单字符 CJK、自动拼音/日文读音、模糊和语义搜索；
- 宽查询必须分页和限预算，不能把全部候选搬到 Go；
- 排序协议、规范化和游标都要版本化；
- 旧 `query_publication_id` 需要短期租约，Catalog GC 通过 publication 引用保护两类 revision；
- 进入当前查询依赖的 Overlay，其 control 写入与查询投影发布分离；投影失败时保留旧查询视图并暴露 pending/failed 状态；
- 仅用于当前响应展示的 Overlay 值需要与 snapshot predicate 分离缓存和契约，客户端不能据此本地重排；
- total、排名和高亮仍需在 API 冻结前过门禁，但不改变基础引擎。

## 重新审议条件

真实产品语料证明 FTS5 在召回、排名、语言分析、空间或增量维护上无法达到门禁，且独立索引的 revision 一致性与运维成本可接受时，重新评估 Bleve 或外部引擎。排序协议只有在黄金语料证明基础 Unicode 回退不满足已承诺场景时升级；升级必须使旧游标过期。
