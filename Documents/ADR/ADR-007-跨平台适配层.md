# ADR-007 跨平台适配层

- 状态：接受
- 日期：2026-07-16
- 规范：[文件系统与媒体处理](../规范/08-文件系统与媒体处理.md)、[跨平台与客户端](../规范/09-跨平台与客户端.md)

## 问题

文件身份、路径、Watcher、进程、工具、数据目录和凭据在 Windows、Linux、macOS、容器和网络挂载上差异显著。如果领域和扫描直接调用平台 API，跨平台支持会成为条件分支集合，且编译成功容易被误写为运行支持。

## 决策

跨平台核心只使用相对路径、领域值和抽象端口。FileIdentityProvider、FileWatcher、PathCanonicalizer、ProcessController、ToolDiscovery、AppDirs 和 CredentialStore 由平台适配器实现；桌面选择器、托盘、自启、通知和电源事件属于壳专属端口。

平台支持状态分为可构建、核心已验证、发行候选和正式支持。交叉编译只满足可构建；每个平台和 SMB/NAS 能力按实际运行门禁单独声明。

## 理由

- FileID/dev+inode、大小写、Unicode 和 Watcher 语义无法用一个实现安全抽象；
- 端口隔离允许核心在无桌面、容器和服务模式运行；
- Windows 与 WSL 运行同一核心探针、Linux/macOS 交叉编译，证明边界可行，但也暴露 DrvFS 不能代替 ext4；
- 明确支持级别防止把可构建夸大为正式支持；
- 证据和局限见 [EV-09](../证据/验证记录.md#ev-09-路径和跨平台核心e1--e4e5)。

## 替代方案

- **Windows-first 直接调用平台 API**：短期快，但会把盘符、注册表和 Win32 路径写入领域核心。
- **只用标准库最低公分母**：无法可靠处理文件身份、Watcher 溢出、凭据和系统工具。
- **统一路径大小写折叠**：在大小写敏感文件系统制造身份冲突。
- **Watcher 作为事实源**：事件丢失、合并和溢出会造成永久漂移。
- **把系统能力放进桌面壳后由核心依赖壳**：无壳服务和容器无法运行。

## 影响

- 每个 adapter 必须有契约测试、能力报告和稳定错误映射；
- Watcher 只触发校验，周期扫描负责最终收敛；
- 文件身份线索不进入权威 Blob 身份；
- 正式发布矩阵按目标 OS/文件系统执行，不按 Go `GOOS` 列表自动生成；
- 平台能力缺失必须显式降级，CredentialStore 不可用时不能明文替代。

## 重新审议条件

端口粒度可随真实平台实现调整，但核心与平台边界不撤销。只有产品明确变为单一 OS 且跨平台目标被正式取消时，才可评估合并部分适配；媒体根只读和路径安全仍不变。
