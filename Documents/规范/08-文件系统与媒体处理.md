# 文件系统与媒体处理

> 类型：规范。媒体根只读、路径安全、文件状态和媒体服务以本文为唯一权威来源。

## 只读边界

Source 原媒体根及其子树默认永久只读。所有数据库、状态、日志、缓存、临时文件、缩略图、转码和更新文件必须位于 AppDirs 的独立可写根。

启动时必须对所有 Source 与所有写入根做规范化、真实路径和重叠检查；任一写入根位于 Source 内、Source 位于写入根内，或两个 Source 存在会造成重复索引的重叠时，必须在数据库初始化和扫描前失败。运行中新增 Source 也应用同一校验。

v1 不提供改名、移动、删除或写回 metadata。未来媒体写子系统必须另立 ADR、独立 capability、事务日志、二次确认和恢复设计，不能放宽当前读取路径。

## 路径模型

- Catalog 和 API 内部资源使用 `source_id + 原始相对路径 + 规范比较键`；
- 公共 API 不返回 OS 绝对路径，只有具备受控本地诊断 capability 的接口可显示脱敏位置；
- 路径比较依据 Source 文件系统探测结果，不对所有平台统一大小写折叠；
- mtime 只可作为扫描优化线索，不能作为身份或 ETag；
- FileLocation 状态至少区分 present、offline、missing、moved-candidate 和 inaccessible。

## 安全解析

所有来自 URL、规则、metadata 或用户输入的相对路径必须先完成一次严格解码，再进入唯一安全解析端口。它必须：

1. 拒绝 NUL、绝对路径、盘符、UNC/前导分隔符、`..`、Windows 保留名和尾点/尾空格等危险形式；
2. 使用 Source 的平台 PathCanonicalizer 规范化；
3. 解析 symlink/junction/reparse 后再次确认目标仍在 Source 根内；
4. 检测链接循环和最大深度；
5. 以只读句柄打开，并在平台允许时通过句柄/目录描述符复核目标，降低 TOCTOU；
6. 在错误中只返回稳定 code，不泄露真实路径。

symlink 默认不跟随越出根；“允许跟随”也只能允许根内目标。junction、任意 reparse、SMB 和 openat/Windows handle 复核必须通过平台门禁。

## 文件关联与状态

FileID、dev+inode、路径、大小和快速指纹只用于发现“可能是同一位置或内容”的候选。新文件、跨卷移动或冲突候选通过完整 SHA-256 确认 ContentBlob。实现应避免重复哈希已确认且平台身份线索可靠的未变化文件，但不得因性能将采样指纹提升为权威键。

ContentBlob 与 FileLocation 的权威记录都位于 `catalog.db`，属于可重建的 Catalog revision。ContentBlob 的跨 revision 稳定引用是“哈希算法版本 + 完整 digest”；FileLocation 的稳定引用由 Source ID、位置身份版本和 Source 内位置键组成。`control.db`、CanonicalMedia、Binding、分享或 Overlay 均不得永久保存 Catalog 内部 Blob/FileLocation row ID。

首次登记 ContentBlob 必须完成全内容 SHA-256。需要新 Blob 的 SourceMedia 在哈希成功前只能处于 staging `hash_pending`，不能作为已确认 Blob 进入可发布 revision、媒体读取或固定版本引用。默认情况下，必需媒体的首次哈希 pending/失败会阻塞受影响 Source 的 publication，旧 query publication 继续服务；显式 Source policy 可以隔离并排除非必需媒体、发布结构化 issue，但绝不能以快速指纹或临时 ID 冒充 ContentBlob。

首次哈希是持久、可取消、可重试的 Job/attempt。同一父 Scan Job 内的 completed Hash Job 结果按 Source、规范相对路径、预期 size/mtime、可用身份线索和算法版本复用（同一次扫描内的重试/续算场景）；跨扫描的复用见下方「扫描档案与内容确认」——默认 `incremental` 档案允许按既往已发布 `content_verified` 观察复用，不代表放弃完整 SHA-256 确认，只代表跳过重新读取未变化文件的正文。超大文件和网络盘使用独立有界 I/O 池、带宽/并发预算、单调字节进度和退避，不降低哈希强度；进度按最小字节增量或时间间隔合并，heartbeat 独立更新，终态前强制刷新观测字节。读取前后必须复核同一文件身份、大小和只读句柄，变化时丢弃结果并返回 `CONTENT_CHANGED_DURING_HASH`。重启后可以在安全持久化 SHA-256 状态且重新确认文件未变时续算，否则从头重算；绝不使用未完成 digest。

阶段 3 已将上述完整哈希落为独立 `Hash Job`/attempt、进度、取消、重试和启动恢复代码；当前合成 Correctness 已覆盖临时 Source 的文件变化、取消和结果发布前置条件。真实超大文件、HDD、SMB/NAS、网络挂载、稳定 FileID 和句柄式 TOCTOU 仍属于最终物理 Schema/平台门禁，不能由合成测试替代。

## 扫描档案与内容确认

真实 SSD/HDD 大规模验收（2026-07-19）发现：把“默认扫描”定义为“每次都对全部媒体重新完整哈希”，在真实数十万文件级图库、尤其 HDD 上不可行（单线程约数个文件/秒，二十万+文件级图库粗略估算需要数十小时），不能作为日常扫描路径。因此扫描按 `scanProfile` 分为三种正式档案，完整定义见 [扫描、Catalog 与任务](04-扫描-Catalog与任务.md#扫描档案与内容确认状态)：

- `index`：只发现和定位，不读取媒体正文、不建立 Hash Job，媒体以 `located_unverified` 发布；
- `incremental`（默认）：按 Source、规范化相对路径、大小、mtime 组合证据与既往已发布 `content_verified` 观察比较，一致则复用既往摘要不重新读取正文，不一致或新增媒体建立 Hash Job 完整确认；
- `verify`：忽略既往观察，对本次扫描到的媒体强制重新完整哈希，是本规范原先描述的“可取消的完整性校验 Job”的显式、用户可选形态。

产品状态区分“已定位”（`located_unverified`）、“hash_pending”（已提交 Hash Job、尚未完成）、“已确认”（`content_verified`）和“确认失败/需重试”（Hash Job 失败，SourceMedia 保持前一状态，旧 publication 不受影响）。`located_unverified` 与 `content_verified` 都是可以正式发布的查询状态，不是构建中的半发布 candidate。

`incremental` 复用是性能优化，不改变身份强度：只复用已经通过完整 SHA-256 确认过的既往 digest，绝不用大小、mtime、路径或平台身份新建或伪造 digest。若文件被刻意改成同大小、恢复时间戳且保持采样块不变，`incremental` 扫描不承诺立即发现——这是已知、文档化的权衡，由显式 `verify` 档案负责发现并建立新 Blob。真实平台文件身份（Windows FileID、`dev+inode`）当前未接入（见 [跨平台与客户端](09-跨平台与客户端.md) 的 `FileIdentityProvider` 端口现状），`incremental` 复用目前只依赖路径 + 大小 + mtime 组合证据，是比最终目标更保守的判据集合，接入真实平台身份后可以在不改变发布语义的前提下增强判据可靠性。

无论哪种档案，一次持久化的 observation（size、mtime、location key 与可用身份字段）都必须来自同一次最终定位观察，不得把 discovery 阶段的旧 Stat 与准备发布前重新定位得到的新 Stat 混合；完整规则与 `index`/`incremental` 各自的刷新时机见 [扫描、Catalog 与任务](04-扫描-Catalog与任务.md#扫描档案与内容确认状态)。未显式指定 `scanProfile` 时的默认选择同样取决于 control.db 是否保留该 Source 的持久领域历史，而不只是 Catalog 当前是否有 publication，避免 Catalog 删除重建后被误判为全新 Source；决策表见同节。

网络盘断开或权限暂失时：

- 保留 CanonicalWork、CanonicalMedia、ContentBlob 和 FileLocation 的可用投影；
- 将位置标为 offline/inaccessible，记录结构化原因和最后确认时间；
- 列表仍可显示 metadata 和派生封面；
- 媒体请求返回明确暂不可用状态，而不是伪装成永久 404；
- 恢复连接后通过校验扫描重新确认，不自动创建重复 Canonical 实体。

## 媒体 HTTP

逻辑媒体 URL 使用稳定 CanonicalMedia ID，例如 `/api/v1/media/{media_id}/content`。它表示该 CanonicalMedia 在请求所绑定的 `query_publication_id` 下解析出的有效 ContentBlob，不暴露路径。服务端由该 ID 取得合法 revision 元组；当前内容由 MediaBinding、SourceMedia、来源优先级和可选的用户固定 Blob 稳定引用解析，不是 CanonicalMedia 保存的 Catalog row ID。

媒体读取端点（列出作品媒体、媒体详情、媒体正文 HEAD/GET、按需内容确认 Job、DerivedAsset 创建）都接受可选查询参数 `queryPublicationId` 显式绑定快照：省略为 **current 模式**（读当前 active publication，响应回显实际使用的 ID）；显式提供为 **snapshot 模式**（只从该 publication 解析，不存在或已过期返回 `CURSOR_EXPIRED`，不静默回退到 active）。服务端为显式快照读取建立短期 publication 读取租约（复用既有游标租约表与 GC 保护判据），覆盖请求处理期间防止被 GC 回收。详见 [API、实时协议与安全](07-API-实时协议与安全.md#媒体与-derivedasset-的查询快照绑定)。

- v1 支持标准单区间 Range、`Accept-Ranges`、`Content-Length`、`If-None-Match`、`If-Range` 和 HEAD；多区间响应延后，不在本轮范围；
- ETag 基于 ContentBlob 的算法版本 + 完整 digest，不基于 Media ID、Catalog row ID、路径或 mtime；`If-Range` 与当前 ETag 不一致时忽略 Range 并退回完整响应，避免拼接不同内容版本的字节；
- 原文件可直接流式读取时不复制到缓存；
- `media.download` 与在线读取 capability 分离仍延后，当前统一由 `media.read` 门禁；
- MIME/扩展名由受控探测与规则共同确定，响应必须防 MIME sniffing；
- Blob 在读取期间改变或位置失效时返回稳定冲突/离线 code，不发送拼接内容；`content_verification_state = located_unverified` 的媒体返回专用 `CONTENT_NOT_VERIFIED`，不进入依赖 ContentBlob/ETag 的已验证读取路径，也不与 Source 离线的 `MEDIA_OFFLINE` 混淆。

## 未确认媒体的按需内容确认

`located_unverified` 媒体在读取时可以通过 `POST /api/v1/media/{mediaId}/verification-jobs` 请求按需完整确认，而不是必须等待下一次周期性扫描：

- 返回持久 Job，不在 HTTP 请求内同步阻塞计算完整 SHA-256；
- **只强制该媒体重新完整哈希，不触发所属 Source 的整体 `verify` 档案**：复用既有 Scanner/Hash Job/Catalog publication 管线中的目标化 Scan Job（`verificationTargets`，见 [扫描、Catalog 与任务](04-扫描-Catalog与任务.md#单媒体目标化按需内容确认)），不新建第二套确认引擎，同 Source 其余媒体（包括其余 `located_unverified` 媒体）继续按既有 `incremental` 规则处理，不被这次按需确认强制重新哈希；
- 幂等键按媒体当前观察（媒体 ID、Source、相对路径、size、mtime、内容确认状态、本次解析到的 `queryPublicationId`）派生：文件未变化且针对同一快照的重复请求复用同一 Job，文件已变化或快照已切换则不会复用过期 observation；
- 支持可选 `queryPublicationId`，语义同「媒体 HTTP」一节的 current/snapshot 模式；
- 媒体已是 `content_verified` 时返回结构化 `CONFLICT`，不建立无意义 Hash Job；
- 确认完成后通过正式 Catalog candidate/publication 更新目标媒体，旧 publication 在确认完成前继续可读；失败、取消和服务重启可恢复，语义与「扫描档案与内容确认」一节完全一致；
- 请求时冻结的媒体 ID、相对路径与 observation 指纹不只是持久化，扫描执行阶段必须真正比对：目标文件在排队期间发生的 size/mtime/内容变化、目标从 discovery 中消失（被移动、改名或删除）、目标相对路径实际解析出的 CanonicalMedia 与冻结的媒体 ID 不一致，均使整个确认 Job 失败为稳定结构化 code（`CONTENT_CHANGED_DURING_HASH`、`CONTENT_DISAPPEARED`、`VERIFICATION_TARGET_MISMATCH`），不得静默确认成一个不是用户原本请求的对象；多个目标时任一目标验证失败都不得让 Job 整体成功。

是否支持对 `located_unverified` 媒体直接流式读取原文件仍未决策，默认不引入；当前唯一路径是先按需确认再读取。

## 派生资源

缩略图、预览、转码和波形等 DerivedAsset 的身份为：

```text
blob_algorithm_version + blob_digest
+ transform_id + transform_version + parameters_hash
+ relevant_overlay_input_version_or_hash?
```

- 默认按需生成，可显式安排批量预热；
- 使用 singleflight 避免同一资产重复生成，并按资源类型限并发；
- 生成请求按 ContentBlob（algorithm+digest）内容寻址，不携带 publication 引用；异步执行阶段解析源文件位置时不限定当前 active publication，只要该 digest 在任意仍未被 GC 回收的 revision 中还有可读 occurrence 即为有效输入，创建后 active publication 切换不改变可解析性；创建与执行两个阶段各自为该 digest 建立固定 TTL 的 `media.BlobReadLease`；GC 不只信任这个租约是否仍在有效期内——只要该 digest 仍被任一非终态 DerivedAsset Job（`queued`/`running`/`publishing`，或已失败但尚未耗尽重试次数、仍处于退避等待期的 `failed`）引用，其唯一剩余 occurrence 所在 revision 就继续受保护，不依赖"排队等待、重试退避或单次生成通常不超过租约 TTL"这一假设；Job 进入真正的终态（`completed`、不可重试的永久 `failed`、`cancelled` 或重试耗尽）后保护随之自然解除；

- DerivedAsset Job 的重试沿用既有持久 Job 退避策略（`MaxRetries`/`RetryPolicyJSON`），只对标记为可恢复的失败（例如生成过程中的一次性 I/O 抖动）生效，不放大为对已经不可恢复的输入（Resolver 返回的永久性 `NOT_FOUND` 等）的无意义重试；Resolver 失败时透传其原始结构化错误码与 retryable 语义，不统一改写为笼统的 `DERIVED_ASSET_FAILED`；
- 临时文件写入 AppDirs 后原子替换；取消、超时和失败必须清理临时文件；
- ffmpeg/ffprobe 等外部工具通过 ToolDiscovery 定位并固定允许版本，使用参数数组启动，不拼 shell；
- 不可信容器不在主进程解析；外部进程设置时间、内存、输出和文件大小上限；
- 派生失败不改变 Catalog publication，可单独重试。

DerivedAsset transform resolver 或外部 ToolDiscovery 未配置时属于明确 capability unavailable，必须在创建 Job 前返回稳定错误且不留下 queued/failed Job；当前代码边界不等于已支持 ffmpeg 或完整变换集合。

v1 已实现并公开唯一一条端到端派生路径：受限 JPEG 缩略图（`transformId=thumbnail`，`transformVersion=v1`），通过 `POST /api/v1/media/{mediaId}/derived-assets` 请求、`GET /api/v1/derived-assets/{assetKey}/content`（`assetKey` 是完整 SHA-256 内容寻址 key，不是 Catalog 内部 row ID）读取正文。该 resolver：

- 只解析 JPEG 容器，不解析 PNG/GIF 或任何其它容器，不声称完整 ffmpeg 支持；
- 限制输入文件字节数与解码像素数，超出限额返回结构化 `DERIVED_ASSET_INVALID`，不尝试解码；
- 使用箱式平均在进程内完成等比缩放，不依赖外部工具或额外图像库；
- 输入 ContentBlob 从请求指定（或省略时当前 active）的 `queryPublicationId` 解析并在创建时冻结，不重新从之后可能变化的 active publication 寻找"当前 Blob"代替请求时刻的 Blob；Source/Location 后续变化不改变已创建 DerivedAsset 的内容身份；媒体在该快照下尚未 `content_verified` 时拒绝创建 Job；
- 创建 DerivedAsset Job 需要独立的 `media.derive` capability，与只读的 `media.read` 分离；读取已生成正文仍使用 `media.read`；
- 派生失败不改变 Catalog publication，可单独重试，与本节其余 DerivedAsset 语义完全一致。

完整视频转码、波形和多格式缩略图（PNG/GIF）留待后续阶段接入真实 ffmpeg/ToolDiscovery 后扩展。外部工具实际执行使用 `AppDirs/Temp/jobs/<job_id>/<attempt>/` 所有权目录及 manifest；通用 Temp GC 不按 `*.tmp` 或模糊前缀猜测删除，只清理超过 grace 的终态所有者目录，损坏或无 manifest 的孤儿使用更长 grace。DerivedAsset 缓存仍只由其专用 key、manifest、lease 和 GC 管理。

DerivedAsset 的登记是 `catalog.db` 中可重建、非 publication 核心的缓存元数据，实际字节位于 AppDirs 缓存。只有用户裁切、自定义封面等确实改变输出的 Overlay 输入才进入 key；Favorite、Progress、Note 等无关变化不得造成缓存失效。Catalog 删除后可按稳定 Blob 引用和 control Overlay 输入重建登记与文件。

缓存文件名和伴随 manifest 必须由完整 DerivedAsset key 确定。manifest 至少记录 Blob 算法/digest、transform ID/version、规范化参数 hash、相关 Overlay 输入 hash、输出大小/MIME 和输出校验值；数据库记录丢失后，服务只能在 key 与 manifest 完整校验通过时原子接管现有缓存并重建登记，不能凭文件名或 mtime 猜测复用。manifest 缺失、损坏、版本不匹配或输出校验失败时，把文件视为 orphan/corrupt 并重新生成。

transform、参数规范化、Blob digest 或相关 Overlay 输入任一变化都会产生新 key；旧文件不原地覆盖，留待 GC。GC 使用 mark-and-sweep：活动登记、当前/租约 publication、预热 pin 和活跃读取 lease 为 mark，超过宽限期的旧 key、孤儿 manifest/文件才可 sweep。生成按 key singleflight，写临时文件并原子发布；读取打开不可变文件并持有 lease/句柄，GC 不得删除活跃读取，失败生成也不得影响已有 key。并发请求只能共享同一完成结果或收到稳定 pending/error，不能读到半个文件。

## 媒体类型范围

v1 核心支持浏览器可读取的常见图片、动画和视频原文件，以及旁挂字幕的关联和下载。视频优先原文件 Range；仅在客户端能力不足且策略允许时生成可重建转码。

压缩包、PDF、漫画容器、PSD/CLIP 预览和复杂字幕转换作为 Source adapter 或 DerivedAsset 扩展延后。未知文件仍可作为附件显示和下载，不能仅因不可预览就判定作品损坏。

## 平台门禁

- Windows junction/reparse、长路径和句柄复核；
- Linux 原生 ext4、大小写、inode 复用和 inotify 丢事件；
- macOS APFS、NFC/NFD、FSEvents 和 sandbox 路径；
- SMB/NAS/UNC 断线、重连、身份变化和权限错误；
