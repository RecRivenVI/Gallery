# 文件系统与媒体处理

> 类型：规范。媒体根只读、路径安全、文件状态和媒体服务以本文为唯一权威来源。

## 只读边界

Source 原媒体根及其子树默认永久只读。所有数据库、状态、日志、缓存、临时文件、缩略图、转码和更新文件必须位于 AppDirs 的独立可写根。

启动时必须对所有 Source 与所有写入根做规范化、真实路径和重叠检查；任一写入根位于 Source 内、Source 位于写入根内，或两个 Source 存在会造成重复索引的重叠时，必须在数据库初始化和扫描前失败。运行中新增 Source 也应用同一校验。

v1 不提供改名、移动、删除或写回 metadata。未来媒体写子系统必须另立 ADR、独立 capability、事务日志、二次确认和恢复设计，不能放宽当前读取路径。

## 路径模型

- Catalog 和 API 内部资源使用 `source_id + 原始相对路径 + 规范比较键`；
- 公共 API 不返回 OS 绝对路径，只有具备受控本地诊断 capability 的接口可显示脱敏位置；
- 路径比较依据 Source 文件系统探测结果，不对所有平台统一大小写折叠；
- mtime 只可作为扫描优化线索，不能作为身份或 ETag；
- FileLocation 状态至少区分 present、offline、missing、moved-candidate 和 inaccessible。

## 安全解析

所有来自 URL、规则、metadata 或用户输入的相对路径必须先完成一次严格解码，再进入唯一安全解析端口。它必须：

1. 拒绝 NUL、绝对路径、盘符、UNC/前导分隔符、`..`、Windows 保留名和尾点/尾空格等危险形式；
2. 使用 Source 的平台 PathCanonicalizer 规范化；
3. 解析 symlink/junction/reparse 后再次确认目标仍在 Source 根内；
4. 检测链接循环和最大深度；
5. 以只读句柄打开，并在平台允许时通过句柄/目录描述符复核目标，降低 TOCTOU；
6. 在错误中只返回稳定 code，不泄露真实路径。

symlink 默认不跟随越出根；“允许跟随”也只能允许根内目标。junction、任意 reparse、SMB 和 openat/Windows handle 复核必须通过平台门禁。

## 文件关联与状态

FileID、dev+inode、路径、大小和快速指纹只用于发现“可能是同一位置或内容”的候选。新文件、跨卷移动或冲突候选通过完整 SHA-256 确认 ContentBlob。实现应避免重复哈希已确认且平台身份线索可靠的未变化文件，但不得因性能将采样指纹提升为权威键。

ContentBlob 与 FileLocation 的权威记录都位于 `catalog.db`，属于可重建的 Catalog revision。ContentBlob 的跨 revision 稳定引用是“哈希算法版本 + 完整 digest”；FileLocation 的稳定引用由 Source ID、位置身份版本和 Source 内位置键组成。`control.db`、CanonicalMedia、Binding、分享或 Overlay 均不得永久保存 Catalog 内部 Blob/FileLocation row ID。

首次登记 ContentBlob 必须完成全内容 SHA-256。需要新 Blob 的 SourceMedia 在哈希成功前只能处于 staging `hash_pending`，不能作为已确认 Blob 进入可发布 revision、媒体读取或固定版本引用。默认情况下，必需媒体的首次哈希 pending/失败会阻塞受影响 Source 的 publication，旧 query publication 继续服务；显式 Source policy 可以隔离并排除非必需媒体、发布结构化 issue，但绝不能以快速指纹或临时 ID 冒充 ContentBlob。

首次哈希是持久、可取消、可重试的 Job/attempt，幂等键至少包含 Source、位置稳定引用和开始时观察到的文件身份/大小。超大文件和网络盘使用独立有界 I/O 池、带宽/并发预算、单调字节进度和退避，不降低哈希强度；耗时增长只延后 publication。读取前后必须复核同一文件身份、大小和只读句柄，变化时丢弃结果并返回 `CONTENT_CHANGED_DURING_HASH`。重启后可以在安全持久化 SHA-256 状态且重新确认文件未变时续算，否则从头重算；绝不使用未完成 digest。

阶段 3 已将上述完整哈希落为独立 `Hash Job`/attempt、进度、取消、重试和启动恢复代码；当前合成 Correctness 已覆盖临时 Source 的文件变化、取消和结果发布前置条件。真实超大文件、HDD、SMB/NAS、网络挂载、稳定 FileID 和句柄式 TOCTOU 仍属于最终物理 Schema/平台门禁，不能由合成测试替代。

普通增量扫描可以根据平台身份和变化线索跳过已确认内容，但这只是性能优化；另行提供可取消的完整性校验 Job，对既有 Blob 重新全量哈希。若文件被刻意改成同大小、恢复时间戳且保持采样块不变，普通扫描不承诺立即发现，完整性校验必须能发现并建立新 Blob。产品状态应区分“已定位”“hash_pending”“已确认”和“确认失败/需重试”。

网络盘断开或权限暂失时：

- 保留 CanonicalWork、CanonicalMedia、ContentBlob 和 FileLocation 的可用投影；
- 将位置标为 offline/inaccessible，记录结构化原因和最后确认时间；
- 列表仍可显示 metadata 和派生封面；
- 媒体请求返回明确暂不可用状态，而不是伪装成永久 404；
- 恢复连接后通过校验扫描重新确认，不自动创建重复 Canonical 实体。

## 媒体 HTTP

逻辑媒体 URL 使用稳定 CanonicalMedia ID，例如 `/api/v1/media/{media_id}/content`。它表示该 CanonicalMedia 在请求所绑定的 `query_publication_id` 下解析出的有效 ContentBlob，不暴露路径。服务端由该 ID 取得合法 revision 元组；当前内容由 MediaBinding、SourceMedia、来源优先级和可选的用户固定 Blob 稳定引用解析，不是 CanonicalMedia 保存的 Catalog row ID。

- v1 支持标准单区间 Range、`Accept-Ranges`、`Content-Length`、HEAD 和条件请求；多区间响应是否需要在 API 门禁中另行确定；
- ETag 基于 ContentBlob 的算法版本 + 完整 digest、响应变换版本和必要的授权变体，不基于 Media ID、Catalog row ID、路径或 mtime；
- 原文件可直接流式读取时不复制到缓存；
- `media.download` 与在线读取 capability 可分离；
- MIME/扩展名由受控探测与规则共同确定，响应必须防 MIME sniffing；
- Blob 在读取期间改变或位置失效时返回稳定冲突/离线 code，不发送拼接内容。

## 派生资源

缩略图、预览、转码和波形等 DerivedAsset 的身份为：

```text
blob_algorithm_version + blob_digest
+ transform_id + transform_version + parameters_hash
+ relevant_overlay_input_version_or_hash?
```

- 默认按需生成，可显式安排批量预热；
- 使用 singleflight 避免同一资产重复生成，并按资源类型限并发；
- 临时文件写入 AppDirs 后原子替换；取消、超时和失败必须清理临时文件；
- ffmpeg/ffprobe 等外部工具通过 ToolDiscovery 定位并固定允许版本，使用参数数组启动，不拼 shell；
- 不可信容器不在主进程解析；外部进程设置时间、内存、输出和文件大小上限；
- 派生失败不改变 Catalog publication，可单独重试。

DerivedAsset 的登记是 `catalog.db` 中可重建、非 publication 核心的缓存元数据，实际字节位于 AppDirs 缓存。只有用户裁切、自定义封面等确实改变输出的 Overlay 输入才进入 key；Favorite、Progress、Note 等无关变化不得造成缓存失效。Catalog 删除后可按稳定 Blob 引用和 control Overlay 输入重建登记与文件。

缓存文件名和伴随 manifest 必须由完整 DerivedAsset key 确定。manifest 至少记录 Blob 算法/digest、transform ID/version、规范化参数 hash、相关 Overlay 输入 hash、输出大小/MIME 和输出校验值；数据库记录丢失后，服务只能在 key 与 manifest 完整校验通过时原子接管现有缓存并重建登记，不能凭文件名或 mtime 猜测复用。manifest 缺失、损坏、版本不匹配或输出校验失败时，把文件视为 orphan/corrupt 并重新生成。

transform、参数规范化、Blob digest 或相关 Overlay 输入任一变化都会产生新 key；旧文件不原地覆盖，留待 GC。GC 使用 mark-and-sweep：活动登记、当前/租约 publication、预热 pin 和活跃读取 lease 为 mark，超过宽限期的旧 key、孤儿 manifest/文件才可 sweep。生成按 key singleflight，写临时文件并原子发布；读取打开不可变文件并持有 lease/句柄，GC 不得删除活跃读取，失败生成也不得影响已有 key。并发请求只能共享同一完成结果或收到稳定 pending/error，不能读到半个文件。

## 媒体类型范围

v1 核心支持浏览器可读取的常见图片、动画和视频原文件，以及旁挂字幕的关联和下载。视频优先原文件 Range；仅在客户端能力不足且策略允许时生成可重建转码。

压缩包、PDF、漫画容器、PSD/CLIP 预览和复杂字幕转换作为 Source adapter 或 DerivedAsset 扩展延后。未知文件仍可作为附件显示和下载，不能仅因不可预览就判定作品损坏。

## 平台门禁

- Windows junction/reparse、长路径和句柄复核；
- Linux 原生 ext4、大小写、inode 复用和 inotify 丢事件；
- macOS APFS、NFC/NFD、FSEvents 和 sandbox 路径；
- SMB/NAS/UNC 断线、重连、身份变化和权限错误；
- 稀疏文件、硬链接、symlink 循环、跨卷移动和快速指纹碰撞；
- Docker 只读媒体挂载与非 root AppDirs；
- 恶意/超大媒体的资源限制和工具崩溃恢复。

身份模型见 [领域模型与数据所有权](03-领域模型与数据所有权.md)，平台支持级别见 [跨平台与客户端](09-跨平台与客户端.md)。
