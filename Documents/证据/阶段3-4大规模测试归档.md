# 阶段 3-4 大规模测试归档：1M/10M 合成语料的方法、过程、结果与局限

> 类型：证据（一次性大规模测试的权威归档）。本文档记录 2026-07-21～2026-07-23 期间在本机 SSD/HDD
> 测试根上执行的 100 万（1M）与 1000 万（10M）规模合成 Catalog 测试的完整方法、真实结果和局限，
> 并说明为什么正式验证目标随后调整为 500,000 WorkProjection（见 [验证记录 EV-35](验证记录.md)、
> [测试与发布门禁](../指南/02-测试与发布门禁.md)）。本文档写成后，对应的本地原始测试产物
> （AppDirs、数据库、WAL、日志、manifest、report）已按 [测试与发布门禁](../指南/02-测试与发布门禁.md)
> 定义的安全协议清理，仅本文档记录的数字、结论和判断依据保留。

## 一、测试目的

验证阶段 4「查询和媒体」的查询、排序、Total、游标、Overlay 依赖集在真实规模（而非单元测试的
小型合成夹具）下的正确性（Correctness）与延迟表现（Reference Performance），为阶段 4 的 API
冻结提供事实依据，同时探索千万级规模下 Catalog publication 流水线（`Stage` →
`ApplyCatalogCandidateOverlays` → `ValidateCandidate` → `Publish`）本身的可扩展性。

## 二、测试框架版本

- 使用的测试工具：`tools/stage4/**`（当时仍是仓库工作树中的未跟踪文件，未提交、未推送）。
- 组成：`tools/stage4/seed`（合成 Catalog 构建器，直接调用生产 `internal/catalog.Store`）、
  `tools/stage4/probe`（HTTP 黑盒驱动器，只导入 `pkg/galleryapi` 生成客户端与标准库，编译并启动
  真实 `cmd/galleryd` 二进制）、`tools/stage4/internal/corpus`（确定性合成语料生成的纯函数）。
- 仓库基线提交：`f80c810dff1999986ddb51269af4dbf85e0cfd82`。

## 三、数据生成方法

`stage4seed` 通过生产 `internal/catalog.Store` 序列，以固定确定性规则批量生成合成
`WorkFact`/`MediaFact`/`OverlayFact`（标题、创作者、标签、文件名均由 `corpus` 包的纯函数按索引
`i` 确定性派生，不依赖真实媒体内容），按 `BatchSize`（默认 20000，故意与创作者槽位周期不对齐）
分批 `Stage`，随后整体 `ApplyCatalogCandidateOverlays`、`ValidateCandidate`、`Publish`，最终产出
一个合法 `query_publication_id` 与描述规模、耗时、创作者 ID 列表的 manifest JSON。所有数据为
纯合成结构化事实（数字化标题/标签/创作者名，不含任何真实用户数据或真实媒体内容）。

## 四、运行命令（去敏形式）

```text
stage4seed  -approot <SSD 测试根>/stages/stage4/appdirs/query-1m  -scale 1000000  -manifest-out <...>/query-1m.json
stage4probe -scenario correctness -approot <...>/query-1m -manifest <...>/query-1m.json -results-out <...>/query-1m-correctness.json
stage4probe -scenario cursor      -approot <...>/query-1m -manifest <...>/query-1m.json -results-out <...>/query-1m-cursor.json
stage4probe -scenario perf        -approot <...>/query-1m -manifest <...>/query-1m.json -results-out <...>/query-1m-perf.json -perf-matrix=full

stage4seed  -approot <SSD 测试根>/stages/stage4/appdirs/query-10m -scale 10000000 -manifest-out <...>/query-10m.json
（10M 构建在 120 分钟硬上限处被外部脚本强制终止，未执行后续 stage4probe 调用）
```

## 九、1M 性能结果与慢查询数据

`query-1m-perf.json`（生成时间 2026-07-22T00:26:16Z，schemaVersion 2）：全部
`perf/*-no-failed-runs` 断言为 `pass=true`（即全部请求最终成功返回、无网络错误或超时），但
`failures=0` **不代表性能达标**——该 finding 只检查请求是否成功返回，与延迟数值无关（这是测试
设计上正确性与性能的刻意分离，见第十一节）。真实延迟分位数：

| 查询类别 | 并发 1 P50 | 并发 16 P50 | 并发 16 P95 | 并发 16 P99 |
| --- | --- | --- | --- | --- |
| `browse`（无过滤无搜索） | 亚秒级 | 亚秒级 | 亚秒级 | 亚秒级 |
| `selective-cjk`（高选择性中文查询） | 低秒级 | 低秒级 | 低秒级 | 低秒级 |
| `wide-cjk`（宽泛中文搜索，limit 20） | 约 9.9-24.3 秒（按 limit 不同） | **约 73.4 秒** | **约 121.4 秒** | **约 143.7 秒** |
| `structured-and`/`structured-or`（含 tag 的结构化组合过滤） | 约 7-10 秒（单线程） | 约 44-121 秒区间（与 wide-cjk 同量级） | 同左 | 同左 |
| `overlay-favorite` | 与上述结构化过滤同量级 | 同左 | 同左 | 同左 |

`structured-and`/`structured-or`/`overlay-favorite` 在并发 16 下的具体分位数与 `wide-cjk` 处于
同一数量级（44-121 秒区间），完整逐组合数值保留在已归档的 `query-1m-perf.json` 原始文件中
（42,494 字节，随本轮清理一并删除，见第十五节"删除后无法再取得的原始证据"）。

### 根因（只读源码审查，未修改任何生产代码）

- `internal/query/service.go` 的排序 CTE（`tiers`/`scored` 两层）在应用 `LIMIT` 前必须对全部
  `WHERE` 匹配候选逐行计算 `title_tier`/`creator_tier`/`tag_tier`/`filename_tier` 四个 `match_class`
  字段并整体排序——这是"先候选、后排序、最后截断"架构在宽查询下的固有代价，不是缺失索引的
  疏漏。对覆盖率约 90% 的宽查询词，候选量接近 90 万行都要参与排序。
- `provider.id`/`tag` 过滤使用逐行关联子查询（`CORRELATED SCALAR SUBQUERY`），`tag` 过滤额外
  对 `tags_json` 做 `json_each` 虚拟表扫描，是含 tag 组合过滤延迟最高的直接原因。
- `internal/storage/database.go` 的 `db.SetMaxOpenConns(4)`/`SetMaxIdleConns(4)` 同时应用于
  `control.db` 与 `catalog.db`（复用同一 `openDatabase` 函数），无论 HTTP 并发多高，真正能同时
  执行 SQL 的连接最多 4 个，其余请求在连接池队列中等待。并发 16 下延迟约等于"单查询成本 ×
  约 4 倍排队系数"，与连接池上限 4 高度吻合。
- `computeTotal` 的 total 查询本身是有界的（`LIMIT TotalBudget+1=10001`），命中数超预算即提前
  停止扫描；上述延迟主要来自结果排序页查询本身无法被 `LIMIT` 提前截断，而非 total 预算扫描。

以上四项统一归类为 `PERFORMANCE_FINDING`（性能候选发现，非本轮修复的生产缺陷），仅登记、
不实施优化，优化候选清单见第十四节。

## 十、完整 1M 性能矩阵耗时的事实与证据边界

1M cursor 报告生成于 2026-07-21T22:49:04Z，perf 报告生成于 2026-07-22T00:26:16Z，两个报告
生成时间戳相差约 **97 分钟**。这是根据"cursor 场景结束时间"到"perf 场景结束时间"的时间戳差
推算出的矩阵运行时长上界，不是由测试框架显式记录的 `startedAt`/`finishedAt` 字段测得的精确值
——本次 perf 报告使用的 schemaVersion 2 尚不含这两个字段（这两个字段是本轮测试框架修复新增的，
见第十三节第 5 项，修复后生成的报告才会有精确的 `startedAt`/`finishedAt`）。因此"约 97 分钟"
是一个有事实依据但存在若干分钟误差的推算值，不应解读为逐组合累加的精确耗时。

## 十一、为什么 "84/84 断言通过" 不能代表性能门禁通过

`stage4probe` 的每条 `perf/*-no-failed-runs` finding 只检查该批请求是否都成功返回且没有超时/
网络错误，与延迟数值完全无关——这是测试设计上正确性与性能的刻意分离（正确性关心"结果对不
对"，性能关心"多快"），不是工具缺陷。1M 完整矩阵中 `wide-cjk`/`structured-and`/`structured-or`/
`overlay-favorite` 在高并发下延迟达到几十秒到两分钟，但只要请求最终成功返回 200，对应 finding
仍是 `pass=true`。真正的性能结论必须逐条查看延迟分位数，不能用 `failures=0` 代替。

## 十二、10M 构建结果与中止过程

- **结果：因 120 分钟硬上限被主动中止，不代表产品失败，也不代表测试通过。**
- 实际运行时长：约 120 分 45 秒（超过上限后立即执行受控终止）。
- 终止方式：`taskkill /PID <pid>`（不带 `/F`）被系统拒绝，回退为 `taskkill /PID <pid> /F`；终止后
  确认进程已退出，且没有残留的 `stage4seed`/`stage4probe`/`stage4-galleryd` 进程。
- manifest 文件（`query-10m.json`）**从未生成**：`BeginCandidate → Stage →
  ApplyCatalogCandidateOverlays → ValidateCandidate → Publish` 序列在中止时未完成，没有产生
  合法的 `query_publication_id`，因此**没有可查询的已发布快照，也没有任何 publication**。
- 中止时 SSD 剩余空间约 477GB（仍然充足，不是磁盘空间耗尽导致的中止）。
- **未删除任何部分数据库、未清理 AppDirs、未重启构建**：原始半提交状态在归档文档完成前
  一直原样保留，供后续复核。

## 十五、测试工具问题与产品问题的区别

**测试工具问题**（已在本轮测试框架内修复，不涉及生产代码）：

1. `stage4probe` 的 perf 场景此前缺少三层超时（单请求/单组合/整体场景），已修复为默认单请求
   30s、单组合 5 分钟、整体矩阵 30 分钟，可通过命令行参数覆盖。
2. 缺少方向性精简矩阵：新增 `-perf-matrix=directional`，已知慢查询类别不再重复验证已证实的
   高并发退化，用于千万级等已知会命中已证实退化路径的规模。
3. 缺少运行进度输出：每完成一个 `(类别,limit,并发)` 组合即向 stderr 打印进度。
4. Partial report 此前非原子写入：`Report.Save` 已改为"先写临时文件、`fsync`、再 `os.Rename`
   原子替换"，保证中途查看的文件始终是完整可解析的 JSON。
5. 报告缺少 `startedAt`/`finishedAt`/`plannedCombinations`/`completedCombinations`/
   `abortedByTimeLimit`/`abortReason` 字段（`query-1m-perf.json` 生成于修复前，因此不含这些
   字段，见第十节）。
6. 超时/取消此前可能被静默丢弃：已修复为超时与"截止时间已过而未派发"的请求都计入
   `failedRuns`（区分 `timedOut`/`notAttempted`），不会只用成功样本计算分位数。
7. 部分完成的矩阵此前可能被误读为整体通过：已修复为矩阵因超时中止时额外产生一条
   `pass=false` 的 `perf/matrix-completed-without-time-abort` finding。

**产品问题**（生产代码 `internal/query`/`internal/catalog`/`internal/storage` 中的真实性能候选
发现，本轮只登记为 `PERFORMANCE_FINDING`，不修复、不改动生产代码——修复属于阶段 4 正式性能
优化任务，不在本轮范围内）：

1. 排序 CTE 无法在候选阶段提前截断（"先候选、后排序、最后截断"架构的固有代价）。
2. `provider.id`/`tag` 过滤使用逐行关联子查询/JSON 扫描，缺少支持结构化过滤的索引或关系表。
3. `db.SetMaxOpenConns(4)`/`SetMaxIdleConns(4)` 同时限制 `control.db` 与 `catalog.db` 的真实并发。
4. `ApplyCatalogCandidateOverlays` 对整个 revision 做一次性全量 `UPDATE`，且需要一次性把全部行
   的 base 字段读入内存，在千万级规模下可能导致构建耗时远超 120 分钟。

两者的区别是本文档反复强调的边界：**第 1-7 项是测试工具本身的缺陷，已经修复；第 1-4 项（产品）
是通过只读源码审查得到的性能候选发现，尚未修复，也不因为"发现了"就等同于"已经确认是必须
修的架构缺陷"**——它们是本轮测试提供的方向性证据，供未来阶段 4 正式性能优化任务参考。

## 十六、为什么 ≥1,000,000 规模调整为非推荐诊断场景

1. 10M 规模仅"构建并发布一个千万级 revision"这一步就可能需要显著超过 120 分钟，且构建期间
   峰值 AppDirs 体积达约 91.47 GiB，对本机测试环境的时间与磁盘预算而言成本过高，不适合作为
   常规、可重复运行的正式验证目标。
2. 1M 规模虽然构建和 Correctness/Cursor 全部通过，但 Reference Performance Gate 在 `wide-cjk`/
   `structured-and`/`structured-or`/`overlay-favorite` 等类别的高并发场景下明确未通过（延迟达
   44-143 秒），且完整性能矩阵本身耗时约 97 分钟，同样过长，不适合作为每次发布前都要求跑完
   的标准 Gate。
3 与之相对，500,000 WorkProjection（10 个目标来源均摊各 50,000）的构建、Correctness 和参考
  性能验证可以在合理时间和磁盘预算内完成（见 [验证记录 EV-35](验证记录.md)），因此被采纳为新
  的推荐正式验证规模；1,000,000 及以上保留为显式启用的非推荐诊断场景，用于探索性能边界，
  不构成标准发布 Gate 的通过条件。
4. 以上调整不代表 Gallery 产品架构"确定无法支持"千万级规模——第十五节列出的产品性能候选发现
  尚未验证修复效果，10M 中止是"当前实现在给定时间预算内未验证通过"，不是"证明了不可能"。
