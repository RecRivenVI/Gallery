# ADR 索引

本文件是 Gallery 架构决策状态的唯一入口。ADR 正文解释“为什么”，规范定义“必须怎样做”；两者冲突时应先修正文档，不能由实现自行选择。

## 状态定义

- **接受**：证据和边界足够，可按决策实施；只有重新审议条件成立才重开。
- **有条件接受**：架构边界可用，但具体实现尚有发布门禁，必须保持可替换。
- **延后**：不进入 v1；触发条件出现后新建 ADR，不得暗中预埋完整子系统。
- **拒绝**：当前目标下明确不采用；重新讨论必须带来新的产品条件或证据。
- **已替代**：只允许出现在当前 ADR 的演进说明中，不保留多个并列的有效正文。

## 被拒绝的默认方案

下列方案已在当前 ADR 的替代方案中记录，不再建立并列决策：微服务和外部队列、以 PostgreSQL 作为默认数据库、路径或采样指纹作为内容身份、generation-delta 作为默认发布、配置内任意脚本、loopback 匿名管理员、客户端本地排序语义、桌面壳直连数据库。

## 修改规则

1. 满足重新审议条件后，优先更新原 ADR；若决策被完全替换，则新增连续编号 ADR，并在索引中只保留新状态。
2. 延期项目被启动时必须新增 ADR，不能直接把索引表改成“接受”。
3. ADR 不保存性能日志全文，只链接 [验证记录](../证据/验证记录.md)。
4. ADR 编号只表示稳定引用，不表示实施顺序；实施顺序见 [v1 实施计划](../指南/01-v1实施计划.md)。
